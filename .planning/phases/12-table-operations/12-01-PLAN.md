# Plan 12-01: Implement Table Operations

<objective>
Add operational table sessions, occupancy tracking, reservations, availability reads, optional Order/Cart table-session linkage, and table-operation Kafka events while keeping Table Context as the owner of table operations and Order Context as the owner of cart/order logic.
</objective>

<assumptions>

- Reservations are staff/admin managed in this phase; public users can read availability but cannot create reservations yet.
- Occupancy has persisted operational state for manual states such as `CLEANING` and `OUT_OF_SERVICE`, while effective occupancy prioritizes active sessions over manual state and reservations.
- Closing a table session accepts an optional next occupancy state. Default is `AVAILABLE`; valid close targets are `AVAILABLE` and `CLEANING`.
- Order/Cart table-session linkage is optional for backward compatibility. Existing `tableId` order flows must keep working.
- Kafka publishing follows the existing after-commit direct producer pattern; no consumers or outbox are added in this phase.

</assumptions>

<tasks>

1. Add Table Operations domain model and stable errors.
   - Define `TableSessionStatus`: `OPEN`, `CLOSED`, `CANCELLED`.
   - Define `TableOccupancyState`: `AVAILABLE`, `OCCUPIED`, `RESERVED`, `CLEANING`, `OUT_OF_SERVICE`.
   - Define `ReservationStatus`: `PENDING`, `CONFIRMED`, `SEATED`, `CANCELLED`, `NO_SHOW`, `COMPLETED`.
   - Add stable table operation errors:
     - `TABLE_SESSION_NOT_FOUND`
     - `TABLE_SESSION_ALREADY_OPEN`
     - `TABLE_SESSION_NOT_OPEN`
     - `TABLE_OCCUPANCY_TRANSITION_INVALID`
     - `TABLE_RESERVATION_NOT_FOUND`
     - `TABLE_RESERVATION_OVERLAP`
     - `TABLE_RESERVATION_STATUS_INVALID`
     - `TABLE_AVAILABILITY_WINDOW_INVALID`
   - Acceptance: operational errors are handled through the existing table exception/global error contract with stable `{ code, message }` responses.

2. Add persistence and repositories.
   - Add `table_sessions` with table id, area/table snapshot fields, opened/closed/cancelled timestamps, status, actor id, party size, note, and optional reservation id.
   - Add `table_occupancy` with table id, state, reason, updated by, and updated at.
   - Add `table_reservations` with table id, area/table snapshot fields, customer name, phone/email, party size, start/end time, status, note, and timestamps.
   - Enforce one active `OPEN` session per table at service level and, where practical for JPA/MySQL compatibility, with repository checks rather than partial unique indexes.
   - Enforce reservation overlap at service level for active statuses `PENDING`, `CONFIRMED`, and `SEATED`.
   - Acceptance: data model remains under `table_context`; no order table fields are moved out of Order Context.

3. Implement `TableOperationService`.
   - `openSession(tableId, request, actorUserId)` validates table orderability, rejects an already open session, snapshots table/area, creates session, and sets occupancy `OCCUPIED`.
   - `closeSession(sessionId, request, actorUserId)` closes an open session and sets occupancy to `AVAILABLE` or `CLEANING`.
   - `cancelSession(sessionId, request, actorUserId)` cancels an open session and releases occupancy.
   - `setOccupancy(tableId, request, actorUserId)` lets staff/admin set manual states such as `AVAILABLE`, `CLEANING`, and `OUT_OF_SERVICE`, but rejects manual `OCCUPIED` when no open session exists.
   - `listOccupancy()` returns area/table catalog data with effective occupancy and active session/reservation summary.
   - Acceptance: opening/closing/cancelling sessions produces deterministic table state transitions.

4. Implement reservation workflows.
   - `createReservation(request, actorUserId)` validates table orderability, party size, time window, and overlap.
   - `updateReservationStatus(reservationId, request, actorUserId)` supports valid transitions:
     - `PENDING -> CONFIRMED | CANCELLED`
     - `CONFIRMED -> SEATED | CANCELLED | NO_SHOW`
     - `SEATED -> COMPLETED`
   - Seating a reservation can open a table session linked to that reservation if no active session exists.
   - Cancelled, no-show, and completed reservations no longer block availability.
   - Acceptance: overlapping active reservations for the same table/time are rejected with `TABLE_RESERVATION_OVERLAP`.

5. Add availability APIs and DTOs.
   - Add request/response DTOs under `TableDtos` or a dedicated `TableOperationDtos`.
   - Expose staff/admin availability and occupancy reads:
     - `GET /admin/tables/occupancy`
     - `GET /admin/tables/availability?from=&to=&partySize=`
   - Expose public availability read:
     - `GET /tables/public/availability?from=&to=&partySize=`
   - Availability should exclude archived/inactive tables, active sessions, `OUT_OF_SERVICE` tables, and active overlapping reservations.
   - Acceptance: public availability is read-only and does not expose internal actor/session metadata.

6. Add staff/admin operation APIs and security.
   - Add APIs:
     - `POST /admin/tables/{tableId}/sessions`
     - `POST /admin/table-sessions/{sessionId}/close`
     - `POST /admin/table-sessions/{sessionId}/cancel`
     - `PATCH /admin/tables/{tableId}/occupancy`
     - `POST /admin/tables/reservations`
     - `PATCH /admin/tables/reservations/{reservationId}/status`
     - `POST /admin/tables/reservations/{reservationId}/seat`
   - Allow both `ADMIN` and `STAFF` for table operation APIs.
   - Preserve existing admin table catalog behavior.
   - Acceptance: ordinary `USER` cannot mutate sessions, occupancy, or reservations.

7. Add optional Order/Cart table-session linkage through ports.
   - Add an Order Context port, for example `TableSessionValidationPort`, returning a stable table-session snapshot when a session id is supplied.
   - Add optional `tableSessionId` to cart item add requests and stored cart/order snapshots.
   - Keep current `tableId` flow valid when no `tableSessionId` is supplied.
   - When a session id is supplied, validate that the session is open and belongs to the same table.
   - Submitted orders copy the optional session id into submitted order persistence and responses.
   - Acceptance: Order Context depends only on a port/snapshot, not Table Context repositories or entities.

8. Add table operation events.
   - Define event payloads:
     - `TableSessionOpened`
     - `TableSessionClosed`
     - `TableOccupancyChanged`
     - `ReservationCreated`
     - `ReservationStatusChanged`
   - Add `TableOperationEventPublisher` port and Kafka adapter/config.
   - Publish after transaction commit using the same pattern as Order/Payment contexts.
   - Acceptance: no Kafka consumers are introduced.

9. Add focused unit tests.
   - Session tests:
     - open session sets occupancy `OCCUPIED`.
     - duplicate open session is rejected.
     - close session defaults to `AVAILABLE`.
     - close session can set `CLEANING`.
   - Reservation tests:
     - overlapping active reservation is rejected.
     - terminal reservations do not block future reservations.
     - invalid status transitions are rejected.
   - Availability tests:
     - active sessions, active reservations, inactive tables, and `OUT_OF_SERVICE` tables are excluded.
   - Order-link tests:
     - supplied session id must be open and match table id.
   - Acceptance: unit tests cover core transition rules and stable error codes.

10. Add integration tests.
   - Admin/staff can open/close/cancel sessions through HTTP.
   - User cannot mutate table operations.
   - Public availability returns only available active tables.
   - Reservation create/status/seat flow works and seating creates or links an open session.
   - Cart/order submission still works without `tableSessionId`.
   - Cart/order submission works with a valid `tableSessionId` and submitted order response includes it.
   - Mock table operation event publisher; no real Kafka broker required.
   - Acceptance: Phase 08 table catalog, Phase 09 cart, Phase 10 order submission, and Phase 11 payment tests still pass.

11. Run full Maven tests.
   - Command: `mvnw.cmd test` with Java 17.

</tasks>
