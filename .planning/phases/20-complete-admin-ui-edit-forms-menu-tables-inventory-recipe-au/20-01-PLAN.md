## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin-ui client → Spring Boot API | All bindings cross here carrying the Bearer JWT; server is the authority. |
| JWT payload → client role state | The `roles` claim is decoded read-only for display/gating; its signature is NOT verified client-side. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2001-01 | Spoofing / Elevation | `decodeRoles` in auth.ts | accept | Client trusts the claim only for UI hiding; the JWT was already verified server-side by JwtAuthenticationFilter over the authenticated session, and every real authorization decision stays with SecurityConfig. `isAdmin` is never used as an authorization gate, only a rendering hint. |
| T-2001-02 | Denial of Service (self) | `sessionsApi.revokeOthers` | mitigate | Read `authState.refreshToken` live at call time inside the binding (not a stale closure) so the current device is correctly excluded from revocation. |
| T-2001-03 | Tampering (supply chain) | npm installs | accept | Zero new packages installed this phase (RESEARCH §"Package Legitimacy Audit" — not applicable); role decode is hand-rolled atob, no new dependency surface. |
</threat_model>
