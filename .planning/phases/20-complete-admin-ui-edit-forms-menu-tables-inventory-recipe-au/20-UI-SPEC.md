# Phase 20 ‚Äî UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase EXTENDS the existing Vue 3 + Vite admin app shipped in Phase 19 (`admin-ui/`). No new design system is introduced. Every token, color, spacing value, and component pattern below was extracted by inspecting `admin-ui/src/style.css` and the existing component primitives ‚Äî this is a record of the contract that ALREADY EXISTS, plus the interaction contracts for the six NEW UI surfaces this phase adds. New UI must reuse these values exactly; it must not invent new tokens.

| Stack | Vue 3.5, Vite 8, vue-router 5, vitest ‚Äî no state library (module-level `reactive`/`ref` singletons, e.g. `stores/auth.ts`) |

## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|

| xs | 6px | `.tag-chip` internal gap; small icon-button internal padding contexts |
| sm | 8px | Button/control internal gap (`.primary-button`, `.ghost-button`), `.nav-list` gap, input vertical padding |
| sm+ | 10px | Input/select/textarea horizontal padding (`padding: 8px 10px`) |

| xl | 24px | `.panel`/`.modal-panel`/`.login-panel` padding, desktop `.content` padding, `.page-heading` gap |

All six new surfaces (shared edit modal, recipe builder, topping management, sessions view, reservation status modal, role-gated controls) must compose exclusively from these values ‚Äî e.g. a new "recipe row" component uses `padding: 8px 10px`, `gap: 12px`, `border-radius: 6px` (matching `.tag-chip`/inputs), not a new radius or padding number.

## New UI for Phase 20 ‚Äî Interaction Contracts

### 1. Shared create+edit modal (D-01)

Applies to 5 entities: Menu category, Menu dish, Table area, Table, Inventory ingredient. Backend confirms `PUT` exists for all five (`AdminMenuController.updateCategory/updateDish`, `AdminTableController.updateArea/updateTable`, `InventoryController.updateIngredient`).

- **Component contract:** reuse `Modal.vue` unmodified (`open`, `title` props; `close` emit). Each entity keeps ONE form component/inline `<form class="field-grid">` that now serves both create and edit ‚Äî do not fork into two modals per entity.
- **Trigger:** `open{Entity}Modal(row?: EntityResponse)`. When called with no argument ‚Üí create mode (form reset to defaults, as today). When called with a row ‚Üí edit mode: seed every reactive form field directly from the row's current values (same field set, exact types ‚Äî `v-model.number` for numeric fields, literal union for `status`/enum selects), and track an `editing{Entity}Id` ref set to the row's id.
- **Title convention (sentence case, matches existing lowercase-entity-name idiom):** create = `New {entity}` (unchanged, e.g. "New category"), edit = `Edit {entity}` (e.g. "Edit category", "Edit dish", "Edit dining area", "Edit table", "Edit ingredient").
- **Layout:** unchanged ‚Äî `.field-grid` two-column grid, `.span-2` for full-width fields, identical field order/labels to the existing create form for that entity.
- **Validation display:** unchanged ‚Äî `.form-error` paragraph (`span-2`), populated via `messageOf(caught, fallback)`, shown directly above `.modal-footer`.
- **Actions:** unchanged `.modal-footer` ‚Äî `.ghost-button` "Cancel" (closes without saving) + `.primary-button` submit, label toggles `Save changes` / `Saving‚Ä¶` while pending, IDENTICAL between create and edit.
- **Submit routing:** if `editing{Entity}Id` is set ‚Üí call `{module}Api.update{Entity}(id, payload)`; else ‚Üí existing `create{Entity}` call. On success in either branch: close modal, reset the editing id, reload the list (`loadMenu`/`loadAll`/`loadStock`).
- **Entry point:** add an "Edit" `ghost-button small` to each row's `.table-actions` cell, positioned BEFORE the existing "Archive" button (non-destructive action precedes destructive, matching the left-to-right severity ordering already used in the reservations row: Seat before Cancel).
- **Category/area chips:** categories and dining areas render as `.tag-chip` lists today (not `DataTable` rows) ‚Äî clicking the chip TEXT (not the √ó remove control) opens edit mode for that chip; the √ó stays wired to archive, unchanged.

### 2. Recipe builder ‚Äî dedicated screen (D-02)

New child route of `AdminLayout`, e.g. `/menu/dishes/:dishId/recipe`, reached via a new "Recipe" `ghost-button small` on each dish row in `MenuView`'s dish table (`table-actions` cell, after Edit/Archive). Not a modal ‚Äî repeatable nested rows are out of scope for the shared modal per D-02.

- **Page heading:** eyebrow "Admin module" ‚Üí h2 `Recipe ‚Äî {dishName}` ‚Üí paragraph "Ingredients required to prepare this dish, and its computed cost." A `ghost-button` "Back to menu" returns to `/menu` without saving.
- **Load:** on mount, fetch existing recipe via `menuApi.getRecipe('DISH', dishId)` (pre-fill rows) and ingredient options via `inventoryApi.listIngredients()`.
  - Loading: `.skeleton-table` block (existing shimmer), same as other panels.
  - Error: `EmptyState` `tone="error"` with `Retry`, same convention as every other panel.
  - Empty (0 rows, new dish with no recipe yet): `EmptyState` `tone="empty"` title "No ingredients yet" body "Add ingredient rows to build this dish's recipe. Costs calculate once you save." ‚Äî no retry button.
- **Repeatable row pattern:** each row is a flex container reusing the existing token set ‚Äî `padding: 8px 10px; border: 1px solid var(--border); border-radius: 6px; background: var(--surface); display: flex; gap: 12px; align-items: center;` (same idiom as `.tag-chip`, not a new visual). Row contents, left to right:
  1. Ingredient `<select>` (options from `listIngredients()`, same pattern as the Inventory movement-form ingredient select)
  2. Quantity `<input type="number" step="0.01" min="0">`
  3. Unit `<input>` ‚Äî auto-fills from the selected ingredient's `baseUnit` on change (same `onMovementIngredientChange` pattern already used in `InventoryView`), still editable
  4. Remove `icon-button` (`X` icon, `aria-label="Remove ingredient row"`) ‚Äî same visual as the `.tag-chip` remove control
  - `sortOrder` is NOT exposed as a form field ‚Äî auto-assign by row index on save (planner/executor discretion per CONTEXT; reduces form complexity, matches the "sensible defaults" allowance).
  - Beneath the row stack: `ghost-button` "Add ingredient" appends an empty row (no ingredient selected, quantity 0, unit blank).
  - Coarse validation (matches the app's existing terse single-line convention ‚Äî no per-row inline errors): a single `.form-error` shown if any row is missing ingredient/quantity/unit at submit time: "Every ingredient row needs an ingredient, quantity, and unit."
- **Cost panel:** separate `.panel` titled "Recipe cost", refreshed only after a successful save (not live/per-keystroke) via `GET /admin/menu/recipes/cost` (backend: `InventoryController.recipeCost`, returns `RecipeCostResponse{totalCost, fullyCosted, lines[]}`). Shows: Total cost (`formatMoney`), a `StatusBadge` (`success`/`warning`) labelled "Fully costed"/"Partial" (same success/warning pattern as `MenuView`'s existing costing table), and a `DataTable` of `RecipeCostLineResponse` (columns: Ingredient, Quantity, Unit, Line cost).
- **Primary actions:** page-level (not modal) footer row ‚Äî `.primary-button` "Save recipe" (calls `menuApi.upsertRecipe(...)`, disabled + "Saving‚Ä¶" while pending) and `.ghost-button` "Back to menu".
- **Role gate:** route guarded ADMIN-only (see ¬ß6) ‚Äî the "Recipe" entry button on the dish row is hidden for STAFF, and the route itself redirects STAFF back to `/menu` if deep-linked.

### 3. Topping management ‚Äî first-class section

Location: a new "Topping groups & options" section inside `MenuView.vue`, REPLACING the current placeholder `.notice-panel` ("a dedicated editor UI is deferred‚Ä¶") ‚Äî that placeholder copy must be removed once this ships.

- **No edit/update for toppings in this phase** ‚Äî confirmed no `PUT` endpoint exists for topping groups or options (`AdminMenuController` only has `POST`/`DELETE` for both). Topping groups/options are **create + archive only**; do NOT apply the D-01 shared-modal edit pattern here ‚Äî there is no backend to back it. Also no archive endpoint for groups (only `archiveToppingOption` exists) ‚Äî group deletion is out of scope, documented as a gap if surfaced.
- **Structure:** per dish (reusing the `menu.categories[].dishes[].toppingGroups[]` tree already returned by `getPublicMenu`, no new fetch needed for read), a `.panel`:
  - Header: dish name + `ghost-button small` "New group" (opens `Modal` "New topping group": fields name, `minSelections` number, `maxSelections` number, `sortOrder` number ‚Üí `menuApi.createToppingGroup`)
  - Each topping group renders as its own nested `.panel` (or a lighter sub-block using the same border/radius/padding tokens): header = group name + a `StatusBadge tone="info"` showing `"{minSelections}‚Äì{maxSelections} selections"` + `ghost-button small` "New option" (opens `Modal` "New topping option": fields name, `additionalPrice` number step 0.01, `status` select ACTIVE/INACTIVE/ARCHIVED, `sortOrder` number ‚Üí `menuApi.createToppingOption`)
  - Options render as a `.tag-list` of `.tag-chip`s: `"{name} (+{formatMoney(additionalPrice)})"` with an √ó wired to `menuApi.archiveToppingOption` ‚Äî identical interaction to the existing category-chip archive.
  - Empty (dish has zero groups): inline `.field-hint` "No topping groups yet." (matches "No categories yet."/"No dining areas yet." idiom) ‚Äî not a full `EmptyState` block, since it's nested inside an already-loaded panel.

### 4. Auth session management

New route + nav item (flat nav-list pattern, matching the existing one-route-per-module convention ‚Äî no nested settings menu): add `{ to: '/sessions', label: 'Sessions', icon: <a distinct lucide icon not already used, e.g. ShieldCheck> }` to `AdminLayout`'s `navItems`, and a child route `{ path: 'sessions', name: 'sessions', component: SessionsView }`. Available to ALL authenticated roles (USER/ADMIN/STAFF) ‚Äî `/auth/sessions` is `hasAnyRole("USER","ADMIN","STAFF")` per `SecurityConfig`; no role gating on this nav item.

- **Page heading:** eyebrow "Account" ‚Üí h2 "Sessions" ‚Üí paragraph "Active sign-ins for your account across devices."
- **Toolbar:** `actions` slot = `ghost-button` "Sign out other sessions" ‚Äî visible/enabled only when the session list has more than 1 row (disabled or hidden when ‚â§1, since revoking others with none active is a no-op). Clicking opens `ConfirmDialog` (title "Sign out other sessions", message "This immediately signs out every other active session for your account. This device stays signed in.", `confirm-label="Sign out others"`, `danger`) ‚Üí `POST /auth/sessions/revoke-others`.
- **Table:** `DataTable` columns = Device/user agent (raw string, `overflow-wrap: anywhere` like `.endpoint-card code` ‚Äî no truncation helper needed, it's not a UUID), IP address, Created (`formatDateTime`), Last used (`formatDateTime`), Actions. Rows from `GET /auth/sessions` ‚Üí `AuthSessionResponse[]`.
- **Row action:** `ghost-button small` "Revoke" ‚Üí `ConfirmDialog` (title "Revoke session", message "Sign this device out immediately? They'll need to log in again.", `confirm-label="Revoke"`, `danger`) ‚Üí `DELETE /auth/sessions/{sessionId}`, then reload the list.
- **Known limitation (do not fabricate):** `AuthSessionResponse` carries no "is this the current session" flag ‚Äî do NOT add a "this device" badge; there is no reliable client-side way to correlate the current session's server-side `sessionId` without an added backend field. Leave all rows visually equal.
- **States:** `.skeleton-table` while loading; `EmptyState tone="error"` + Retry on failure ‚Äî same convention as every other panel.

### 5. Full reservation status control (replaces cancel-only)

Location: `TablesView.vue` ‚Üí "Reservations" panel ‚Üí `table-actions` cell. Replaces the current hardcoded `status: 'CANCELLED'` call (`cancelReservation`, currently `TablesView.vue:325-334`) with a full selector using the SAME modal-driven pattern already established for `openSessionModal` (not an inline per-row `<select>`, to stay consistent with how every other multi-state transition in this app is handled).

- **Trigger:** rename the row's "Cancel" `ghost-button small` to "Status" ‚Äî opens `Modal` titled "Update reservation status".
- **Modal body:** `<select>` of the 4 target statuses (`CONFIRMED`, `NO_SHOW`, `COMPLETED`, `CANCELLED`) excluding the row's current status, plus an optional `note` `<input>` (maps to `UpdateReservationStatusRequest.note`). Footer: `ghost-button` "Cancel" / `primary-button` "Update status".
- **Confirmation escalation:** submitting with `CONFIRMED` or `COMPLETED` selected calls `tablesApi.updateReservationStatus` immediately (forward progress, non-destructive, no extra step). Submitting with `CANCELLED` or `NO_SHOW` selected first closes/holds the status modal and opens a `ConfirmDialog` (`danger`) per the exact copy in the Copywriting Contract above, before the API call fires.
- **No change needed:** `StatusBadge`'s `STATUS_TONES` map already covers all target statuses (CONFIRMED=success, NO_SHOW=danger, COMPLETED=success, CANCELLED=danger) and the existing "Seat" button logic for PENDING/CONFIRMED rows is untouched ‚Äî it is a distinct action, not part of this status selector.

### 6. Role-aware affordances (hide, not disable)

**Verified role matrix** (source of truth: `SecurityConfig.java` `authorizeHttpRequests`, matchers evaluated in declared order ‚Äî first match wins). This CORRECTS an overbroad assumption in `20-CONTEXT.md`'s discretion notes, which stated inventory CRUD is ADMIN-only; the actual matcher order grants `/admin/inventory/**` to ADMIN **and** STAFF, ahead of the general `/admin/**` ADMIN-only catch-all:

| Surface | Roles | Gate in UI? |
|---------|-------|-------------|
| Menu catalog CRUD ‚Äî categories, dishes, topping groups/options (create+archive), recipes (PUT/GET, non-cost) | ADMIN only | **Hide** for STAFF |
| Table catalog CRUD ‚Äî areas, tables (`/admin/tables/areas/**`, bare `/admin/tables`, `/admin/tables/{id}`) | ADMIN only | **Hide** for STAFF |
| Table operations ‚Äî occupancy, sessions, reservations (create/status/seat) | ADMIN + STAFF | Visible to both ‚Äî no change |
| Inventory ‚Äî ingredients CRUD, movements, costs, stock/low-stock reads, `/admin/menu/recipes/cost`, `/admin/menu/costing` | ADMIN + STAFF | Visible to both ‚Äî do NOT hide |
| Payments, Kitchen, Orders (`/admin/payments/**`, `/admin/orders/**`) | ADMIN + STAFF | Visible to both ‚Äî unchanged |
| Auth sessions (`/auth/sessions*`) | any authenticated (USER, ADMIN, STAFF) | Visible to all ‚Äî no gate |

- **Decode mechanism:** decode the `roles` claim (a string array, e.g. `["ADMIN"]`) from the JWT access token's payload segment client-side ‚Äî no round-trip needed (`JwtProvider.generateToken` embeds `claim("roles", roles)`). Base64url-decode the middle segment with existing browser APIs (`atob` + URL-safe padding fix), `JSON.parse`, read `.roles`. Add this to `stores/auth.ts` alongside the existing token state; recompute in `setSession()` and `restoreSession()` (the same two lifecycle points that already persist the token pair). Expose `authState.isAdmin` (or an equivalent computed) for views to consume.
- **Pattern:** `v-if`, never `disabled` + tooltip (locked decision). Apply `v-if="authState.isAdmin"` to: Menu's "New category"/"New dish" buttons, category-chip √ó and edit-click, dish row Edit/Archive/Recipe buttons, all topping group/option create+archive controls, Tables' "New area"/area √ó /"New table"/table Edit/Archive buttons. Do NOT gate any Inventory, Table-operations, Payments, Kitchen, Orders, or Sessions control.
- **Route-level guard:** the new Recipe-builder route (`/menu/dishes/:dishId/recipe`) additionally needs a `router.beforeEach` check (`meta: { adminOnly: true }`) redirecting STAFF back to `/menu` if deep-linked directly ‚Äî button-hiding alone doesn't stop URL navigation. No other new routes in this phase need this guard (Sessions is open to all roles; Topping/edit-modal live inside existing views, gated at the button level only).
- **Non-goal:** this is view-layer UX polish only. The backend remains the actual enforcement boundary; a STAFF user hitting a hidden endpoint directly still gets a 403 via the existing generic `ApiError` ‚Üí `.form-error` path ‚Äî no special handling needed beyond what already exists.
