# Plan 14-01: Implement Inventory Stock Management Foundation

<objective>
Add operational stock management to Inventory Context with stock balances, immutable movement records, manual receipts/adjustments/waste, and low-stock reads while preserving existing costing/menu/order/payment behavior.
</objective>

<assumptions>

- Phase 14 uses one default stock location.
- Stock movements are manual/admin-staff operations.
- Automatic recipe-based deduction from orders is deferred until deduction timing is confirmed.
- Existing ingredient and unit conversion behavior from Phase 13 is reused.
- Public menu and customer-facing order/payment APIs do not expose stock data.

</assumptions>

<tasks>

1. Extend Inventory domain model.
   - Add `InventoryMovementType` with receipt, adjustment in/out, waste, and stock count/correction semantics.
   - Add stable errors:
     - `INVENTORY_MOVEMENT_INVALID`
     - `INVENTORY_MOVEMENT_QUANTITY_INVALID`
     - `INVENTORY_STOCK_INSUFFICIENT`
     - `INVENTORY_STOCK_NOT_FOUND`
   - Add low-stock threshold fields to ingredient persistence or a dedicated stock settings model.
   - Acceptance: stock concepts remain inside Inventory Context.

2. Add persistence.
   - Add `inventory_stock_balances` for ingredient stock-on-hand in the default location.
   - Add `inventory_stock_movements` as immutable audit records.
   - Store quantity, unit, converted/base quantity, reason, note, reference type/id, actor id when available, and timestamp.
   - Acceptance: movement history can reconstruct stock changes and current balance can be queried efficiently.

3. Implement stock movement service.
   - Record inbound receipts and adjustment-in movements.
   - Record outbound waste and adjustment-out movements.
   - Record stock count/correction movements.
   - Validate active ingredient, positive quantity, supported unit conversion, and non-negative stock for normal outbound operations.
   - Update stock balance atomically with movement persistence.
   - Acceptance: a failed movement never partially updates balance.

4. Add stock read service.
   - List stock balances with ingredient metadata, current quantity, unit, last movement time, and low-stock state.
   - Read stock for one ingredient.
   - List low-stock ingredients.
   - Acceptance: staff/admin can quickly inspect operational stock health.

5. Add admin/staff APIs.
   - `GET /admin/inventory/stock`
   - `GET /admin/inventory/ingredients/{ingredientId}/stock`
   - `POST /admin/inventory/movements`
   - `GET /admin/inventory/movements`
   - `GET /admin/inventory/low-stock`
   - Acceptance: `ADMIN` and `STAFF` are allowed; ordinary users and anonymous clients are rejected.

6. Preserve existing behavior.
   - Do not add stock fields to public menu.
   - Do not change recipe cost calculation.
   - Do not change cart/order/payment totals.
   - Do not consume Kafka order/payment events in this phase.
   - Acceptance: existing tests continue passing.

7. Add tests.
   - Unit tests for movement validation, conversion, insufficient stock, and balance update.
   - Unit tests for low-stock threshold detection.
   - Controller/integration tests for staff/admin authorization and stock APIs.
   - Regression test that Phase 13 recipe costing still works after stock changes.
   - Acceptance: focused tests plus full Maven suite pass.

8. Run full Maven tests.
   - Command: `mvnw.cmd test` with Java 17.

</tasks>

<verification>

- Full `mvnw.cmd test` passes.
- `INV-012` through `INV-021` are covered by unit and integration tests.
- Public menu, order, payment, and recipe costing contracts remain backward compatible.
- Movement records are immutable and stock balances update atomically.

</verification>

<success_criteria>

- Staff/admin can record receipts, waste, and stock adjustments.
- Staff/admin can inspect stock-on-hand and low-stock ingredients.
- Stock validation prevents accidental negative inventory for normal outbound movements.
- Inventory management remains cleanly separated from order/payment automation until that timing is explicitly chosen.

</success_criteria>
