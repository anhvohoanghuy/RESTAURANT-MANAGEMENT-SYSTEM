<objective>
Build the Inventory-side reservation persistence layer (D-02/D-03/D-09): add a `reservedQuantity` column and a pessimistic-lock query to the stock balance, a per-order `StockReservationEntity` (unique `orderId`, per-ingredient reserved lines for Phase-16 settlement), and an `InventoryProcessedEventEntity` idempotency ledger.

Purpose: The reservation service (Plan 15-03) needs these persistence primitives — the running reserved total, the row-lock for the non-negative invariant, the unique per-order reservation guard, and the eventId ledger. This plan owns only inventory persistence files, so it runs in Wave 1 in parallel with the order-side foundation (15-01).
Output: Modified balance entity + repository; new reservation entity/repository; new processed-events entity/repository.

**Table-naming coordination:** This app is ONE Spring Boot deployment with ONE datasource and ONE JPA persistence unit (MySQL prod, single H2 test schema). The two saga contexts each get a physically distinct ledger table: this plan's Inventory ledger is `inventory_processed_events`; the Order ledger (Plan 15-05) is `order_processed_events`. They must NOT share a physical table name — two `@Entity` classes mapping the same table would fail DDL/startup.
</objective>
