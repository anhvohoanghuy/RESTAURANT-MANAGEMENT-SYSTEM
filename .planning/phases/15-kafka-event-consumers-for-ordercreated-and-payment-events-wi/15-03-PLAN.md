    Create `InventoryStockResultPublisher` in `inventory_context.domain.port` — a one-method interface `void publishStockResult(OrderStockResultEvent event)` mirroring `OrderEventPublisher`. Create `InventoryReservationService` (`@Service @RequiredArgsConstructor`, constructor-injected `InventoryProcessedEventRepository`, `StockReservationRepository`, `InventoryStockBalanceRepository`, `IngredientRepository`, `MenuRecipeCostingPort`, `InventoryStockResultPublisher`). Implement `@Transactional public void onOrderCreated(OrderCreatedEvent event)` following RESEARCH Pattern 3:
    (1) Idempotency guard — pre-check `existsByEventIdAndConsumerName(eventId, "inventory-order-created")` and `reservations.existsByOrderId(orderId)`; if either true, return. Otherwise `save` the ledger row and immediately `flush()` it (see note) — rely on the unique constraint: catch `DataIntegrityViolationException` around the save+flush and treat it as a concurrent duplicate -> return.
    (2) `computeRequired(event)` -> `Map<UUID,BigDecimal>` by resolving DISH + TOPPING_OPTION recipes via `menuRecipeCostingPort.findRecipe`. INFO/D-06 base-unit sourcing: for each recipe line, look up the ingredient's base unit via `ingredientRepository.findById(line.ingredientId()).map(IngredientEntity::getBaseUnit)` and convert `line.quantity()` from `line.unit()` to that base unit with `UnitConverter.convert`, then multiply by the order `line.quantity()` (int). Accumulate per ingredientId. Tolerate missing recipe / null ingredientId / ingredient not found / unconvertible unit as ZERO for that contribution (reuse the costing `uncosted` tolerance pattern — do not re-query recipes differently, do not throw).
    (3) Build a canonical iteration order: sort the required map's ingredient IDs (e.g. by UUID natural order) into a `List<UUID>` and use it for BOTH the availability-check loop and the reserve loop, so every concurrent reserver locks rows in the same order (W-1 deadlock avoidance). For each ingredientId in sorted order, `lockByIngredientAndLocation(ingredientId, DEFAULT_LOCATION)`, compute `available = quantityOnHand - reservedQuantity`, and collect a `Shortfall` when the balance is missing or available < required.
    (4) If no shortfalls, iterate the SAME sorted list, re-lock (or reuse the locked entity) and increment `reservedQuantity` on each balance, then save a `StockReservationEntity.held(orderId, required)`; build a CONFIRMED `OrderStockResultEvent`. Else build a REJECTED event with the shortfalls.
    (5) Publish via a private `publishAfterCommit(...)` copied from `OrderSubmissionService` so the event emits only after commit (D-10). Use a `QUANTITY_SCALE`/`scale()` helper mirroring `InventoryStockService`. Keep all logic in this service — the listener (15-04) is a thin delegate.
    W-2 flush note: because the ledger `@Id` uses `GenerationType.UUID` the insert is NOT auto-flushed until commit, so a concurrent-duplicate constraint violation would otherwise throw at COMMIT (outside this method's try/catch). The explicit `flush()` right after `save` forces the INSERT now, so the `DataIntegrityViolationException` surfaces inside the guarded block and the handler returns cleanly.
  </action>
