# Plan 13-01: Implement Inventory Costing Foundation

<objective>
Add an Inventory Context for ingredient master data and ingredient costs, link recipes to ingredients while preserving existing recipe behavior, and expose admin/staff recipe/menu costing reads without changing public menu, order, payment, or stock behavior.
</objective>

<assumptions>

- Latest effective ingredient cost is enough for Phase 13; FIFO/weighted-average inventory valuation is deferred.
- Costing is read/calculation behavior only. It does not deduct inventory and does not alter sell prices or submitted order totals.
- Recipe lines can remain free-text for backward compatibility, but new/updated recipe lines may include an optional `ingredientId`.
- Unit conversion is scoped to simple deterministic conversions configured per ingredient or supported by base units. Unsupported conversions should be explicit, not silently guessed.
- Admin/staff can read costing data; ordinary users and public endpoints cannot.

</assumptions>

<tasks>

1. Add Inventory Context package and domain model.
   - Create `inventory_context` with domain/application/infrastructure layers following existing DDD package style.
   - Add ingredient status enum, for example `ACTIVE` and `ARCHIVED`.
   - Add `Ingredient` model with id, name, base unit, optional description, status, and timestamps where consistent with local patterns.
   - Add `IngredientCost` model with ingredient id, unit cost, cost unit, effective at, optional source/note, and created timestamp.
   - Add stable errors:
     - `INVENTORY_INGREDIENT_NOT_FOUND`
     - `INVENTORY_INGREDIENT_NOT_ACTIVE`
     - `INVENTORY_COST_INVALID`
     - `INVENTORY_COST_NOT_FOUND`
     - `INVENTORY_UNIT_CONVERSION_UNSUPPORTED`
   - Acceptance: Inventory Context owns ingredient and cost concepts; Menu Context does not persist ingredient cost fields directly.

2. Add persistence and repositories.
   - Add `inventory_ingredients`.
   - Add `inventory_ingredient_costs`.
   - Add optional unit conversion persistence if the implementation needs non-standard unit conversions, for example `inventory_unit_conversions`.
   - Store money as `BigDecimal` with VND-compatible precision.
   - Repository queries should support:
     - find active ingredients by name/search.
     - latest cost for ingredient effective at or before a timestamp.
     - ingredient cost history ordered by effective time.
   - Acceptance: schema works with Spring Data JPA and existing H2/MySQL-friendly patterns.

3. Add ingredient and cost application service.
   - Implement create/update/archive/list ingredient operations.
   - Implement add cost record and list cost history.
   - Validate positive cost, nonblank units, nonblank ingredient names, and active ingredient state.
   - Prevent public access to costing data.
   - Acceptance: admin/staff can maintain costing data without touching Menu Catalog internals.

4. Add admin/staff inventory APIs and security.
   - Add controller endpoints:
     - `POST /admin/inventory/ingredients`
     - `PUT /admin/inventory/ingredients/{id}`
     - `DELETE /admin/inventory/ingredients/{id}` to archive.
     - `GET /admin/inventory/ingredients`
     - `POST /admin/inventory/ingredients/{id}/costs`
     - `GET /admin/inventory/ingredients/{id}/costs`
   - Allow `ADMIN` and `STAFF` for `/admin/inventory/**`.
   - Preserve existing `/admin/**` admin-only fallback for unrelated endpoints.
   - Acceptance: ordinary `USER` receives `403`; anonymous receives `401`.

5. Extend Menu recipe DTO/model for optional ingredient linkage.
   - Add optional `ingredientId` to `RecipeRequest.Line` and `RecipeResponse.Line`.
   - Add nullable `ingredient_id` to recipe line persistence.
   - Preserve existing recipe request compatibility when clients only send `ingredient` text.
   - Existing recipe unit tests should keep passing with minimal updates.
   - Acceptance: recipes can be created both with free-text ingredient lines and with inventory-linked ingredient lines.

6. Add cross-context ports for recipe costing.
   - Add a Menu Context port/service that returns a recipe snapshot for `RecipeTargetType + targetId`, including line ingredient id/name, quantity, and unit.
   - Add an Inventory Context costing service that consumes recipe snapshots through a port rather than reading menu repositories directly.
   - Add an adapter in Menu infrastructure/application boundary as needed.
   - Acceptance: Inventory Context does not depend on Menu JPA repositories/entities directly.

7. Implement unit conversion and cost calculation.
   - Normalize units consistently, for example `g`, `kg`, `ml`, `l`, and `pcs`.
   - Convert recipe line quantity to the ingredient cost unit or base unit.
   - Compute line cost = converted quantity * unit cost.
   - Compute recipe total cost as sum of costed lines.
   - Mark uncosted lines with reason:
     - no `ingredientId`.
     - missing active ingredient.
     - missing current cost.
     - unsupported unit conversion.
   - Acceptance: calculation is deterministic and never invents costs for missing data.

8. Add recipe and menu costing read APIs.
   - Add `GET /admin/menu/recipes/cost?targetType=&targetId=`.
   - Add `GET /admin/menu/costing` to list active dishes with sell price, estimated recipe cost, topping cost where applicable, gross margin amount, and gross margin percent.
   - Include uncosted/missing-cost flags in responses so operators can find incomplete data.
   - Do not add costing fields to `GET /menus/public`.
   - Acceptance: admin/staff can inspect cost and margin; public menu contract remains unchanged.

9. Add focused unit tests.
   - Ingredient validation and archive behavior.
   - Cost record validation and latest-effective-cost selection.
   - Unit conversion success and unsupported conversion errors.
   - Recipe costing with:
     - all lines costed.
     - missing ingredient link.
     - missing cost.
     - unsupported unit.
   - Menu margin calculation.
   - Acceptance: stable errors and calculation math are covered.

10. Add controller/integration tests.
   - Admin/staff can create ingredient and add cost.
   - User cannot access `/admin/inventory/**`.
   - Recipe upsert works with optional `ingredientId`.
   - Recipe costing endpoint returns line costs and total cost.
   - Public menu response does not expose cost fields.
   - Existing cart/order/payment tests remain unaffected.
   - Acceptance: no real Kafka, payment provider, or external network dependency is introduced.

11. Run full Maven tests.
   - Command: `mvnw.cmd test` with Java 17.

</tasks>
