# Phase 17.2: Inventory Settlement Idempotency Hardening - Context

**Gathered:** 2026-07-09
**Status:** Ready for planning
**Source:** Follow-up from Phase 17.1 verification residual note

<domain>
## Phase Boundary

Fix the remaining Phase 16 settlement idempotency anti-pattern: `InventoryReservationSettlementService`
still records the `inventory_processed_events` ledger row through `InventoryLedgerWriter`, whose
`REQUIRES_NEW` transaction commits before the settlement business work.

This creates the same failure class Phase 17.1 closed for four other consumers: a failure after the
ledger commit but before the business transaction commits can permanently mark a Kafka event as
processed while the settlement work rolls back. On redelivery, the eventId pre-check skips the event,
leaving the order line unsettled.

This phase is a narrow backend hardening phase. It must not change the settle-trigger event contract,
reservation arithmetic, kitchen behavior, order status projection, Kafka topic names, or public APIs.
</domain>

<decisions>
## Implementation Decisions

### Settlement idempotency ordering
- **D-01:** Remove `InventoryLedgerWriter` from `InventoryReservationSettlementService`.
- **D-02:** Record the `InventoryProcessedEventEntity` row as the final durable step inside the same
  `@Transactional` settlement method, after recipe resolution, reservation/balance mutation,
  CONSUMPTION movement writes, line-settlement guard write, and optional reservation `SETTLED`
  status flip.
- **D-03:** Do not catch `DataIntegrityViolationException` from the final ledger insert. If a
  concurrent duplicate violates the `(event_id, consumer_name)` unique constraint, let the whole
  settlement transaction roll back; Kafka redelivery will then hit the existing pre-check and skip
  once the winning transaction has committed.
- **D-04:** Keep both existing idempotency guards: eventId ledger pre-check and per-`(orderId,
  orderLineId)` settlement guard. The line-settlement guard remains the business-key protection
  against new-eventId duplicates for the same order line.
- **D-05:** Delete `InventoryLedgerWriter.java` and its dedicated test if no production caller remains.
  Update stale comments that describe `REQUIRES_NEW` as the desired pattern for settlement.

### Verification decisions
- **D-06:** Add/adjust tests that prove the ledger insert is final and same-transaction:
  concurrent duplicate ledger insertion propagates rather than being swallowed, and no settlement
  path calls `InventoryLedgerWriter`.
- **D-07:** Preserve all Phase 16 behavioral tests: deduction, clamp, CONSUMPTION movement,
  `SETTLED`-on-last-line, duplicate eventId skip, duplicate orderLine skip, missing reservation DLT
  behavior, already-settled benign return, and reservation-before-balance lock ordering.

### Scope fence
- **D-08:** Do not introduce a transactional outbox for kitchen settle-trigger/status-changed events
  in this phase.
- **D-09:** Do not touch Payment/Table Kafka producer serializer or payment-history filters in this
  phase; those are separate backlog items.
</decisions>

<specifics>
## Specific Ideas

- Add a private helper in `InventoryReservationSettlementService`:
  `recordProcessed(UUID eventId)` builds `InventoryProcessedEventEntity`, sets eventId,
  `CONSUMER_NAME`, `processedAt=Instant.now()`, and calls `processedEventRepository.save(...)`.
- Call `recordProcessed(eventId)` as the final statement of the successful settlement path.
- Keep early returns for already processed, already settled line, and non-HELD reservation before
  any mutation.
- If a duplicate line-settlement or duplicate ledger unique constraint fires at flush/commit, let
  the exception propagate so the transaction rolls back atomically.
</specifics>
