# Plan 11-01: Implement Payment Checkout Context

<objective>
Add a Payment Context that records manual partial payments and refunds for submitted orders, exposes QR payment request placeholders, enriches order reads with payment summaries, and publishes core payment Kafka events after commit.
</objective>

<tasks>

1. Add Payment Context domain model and DTOs.
   - Define payment methods `CASH`, `BANK_TRANSFER`, `QR_CODE`.
   - Define payment status `UNPAID`, `PARTIALLY_PAID`, `PAID`.
   - Define refund summary status separately from payment status.
   - Define QR request statuses `PENDING`, `CONFIRMED`, `FAILED`, `EXPIRED`, `CANCELLED`.
   - Add stable payment exception codes for invalid amount, overpay, not found, unauthorized owner, missing idempotency key, duplicate/conflicting idempotency, and refund overflow.
   - Acceptance: payment/refund/payment-request concerns are under `payment_context`, not Order Context.

2. Add persistence and repositories.
   - Create `payments`, `payment_refunds`, and QR payment request persistence.
   - Store order id, order owner id snapshot, amount, method, status metadata, idempotency key, actor id, timestamps, and notes/reference fields.
   - Add uniqueness constraints for payment idempotency per order and refund idempotency per payment.
   - Acceptance: retrying the same idempotency key can return existing records.

3. Add ports/adapters across contexts.
   - Payment Context reads submitted order details through an `OrderPaymentLookupPort` with an adapter backed by `OrderRepository`.
   - Order Context enriches order responses through a `PaymentSummaryPort` with a Payment Context adapter.
   - Acceptance: Order Context does not store payment state fields.

4. Implement payment application services.
   - Staff/admin records manual payment with idempotency and no-overpay validation.
   - Staff/admin records refund with idempotency and no-refund-overflow validation.
   - Authenticated order owner creates QR payment request placeholder with provider-ready `paymentUrl`/`redirectUrl` fields but no provider call or auto-confirm.
   - Compute order payment summary from payments/refunds.
   - Provide order-scoped history and global admin/staff history with cursor pagination and basic filters.
   - Acceptance: partial payment status and refund summary match Phase 11 decisions.

5. Add controllers and security.
   - Add order-scoped payment history API.
   - Add staff/admin manual payment and refund APIs.
   - Add owner QR payment request API.
   - Add admin/staff global payment list with cursor pagination.
   - Seed `STAFF` role and allow `STAFF`/`ADMIN` for payment admin APIs.
   - Acceptance: ordinary `USER` cannot record payments/refunds.

6. Add Kafka payment events.
   - Define event payloads for `PaymentRecorded`, `PaymentRefunded`, and `OrderPaymentCompleted`.
   - Add event publisher port and Kafka adapter.
   - Publish after transaction commit.
   - Acceptance: no Kafka consumers are introduced.

7. Add focused tests.
   - Unit tests for payment service idempotency, partial payment status, overpay rejection, refund idempotency, and refund overflow.
   - Integration tests for manual payment, partial/full summary on order reads, user forbidden from staff APIs, QR request placeholder, global cursor pagination, and event publishing through mocked publisher.
   - Acceptance: no real Kafka broker or payment provider required.

8. Run full Maven tests.
   - Command: `mvnw.cmd test` with Java 17.

</tasks>
