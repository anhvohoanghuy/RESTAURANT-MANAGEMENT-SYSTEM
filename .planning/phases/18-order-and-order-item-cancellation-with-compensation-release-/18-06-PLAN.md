<objective>
Add the Kitchen-side compensation consumer (D-7) that closes the cancel-vs-advance race: on each
`OrderCancelledEvent`, for every cancelled `orderLineId` whose kitchen ticket item is still
`QUEUED`, void it so staff can no longer advance it to `PREPARING` (which would otherwise fire a
`SettleTrigger` deducting stock already released). It NEVER touches an item already `>= PREPARING`
(defense-in-depth mirroring the REJECTED/rank-guard idiom). Idempotent via the existing kitchen
processed-events ledger + DLT.

Purpose: Closes CANCEL-08 (kitchen ticket void on cancel, D-7) — the defensive backstop for the
create→confirm→ticket race that Plan 02's synchronous read narrows but cannot fully eliminate.
Depends on Plan 02 (shares `KitchenTicketItemRepository`); consumes the Plan 01 event contract.
Output: appended `KitchenItemStatus.CANCELLED`, a lock-by-orderLineId query,
`KitchenTicketInvalidationService`, listener, consumer config, advance-guard regression, and
`KitchenTicketInvalidationServiceTest`.
</objective>
