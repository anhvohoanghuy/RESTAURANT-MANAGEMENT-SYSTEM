<objective>
Implement the order_context cancellation core: a race-safe cross-context read of kitchen item
status (new order-owned port + kitchen adapter) and `OrderCancellationService` handling both
whole-order cancel and partial item-cancel with a pessimistic order-row lock, customer-ownership
check, the SUBMITTED/PENDING_CONFIRMATION/CONFIRMED window guard, per-line non-PREPARING exclusion,
total recompute, and a same-transaction outbox publish of `OrderCancelledEvent`.

Purpose: This is the heart of the phase — the write side that transitions the order and emits the
compensation event the three Wave-2/3 consumers react to. Closes CANCEL-01 (window guard),
CANCEL-02 (ownership), CANCEL-04 (partial recompute). REST endpoints are Plan 05.
Output: New `KitchenItemStatusPort` + `KitchenItemStatusView` + `KitchenItemStatusAdapter`,
`OrderCancellationService`, cancellation DTOs, and `OrderCancellationServiceTest`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (customer JWT) → order_context | Untrusted orderId/lineId path values; customer must only affect their own order |
| order_context → kitchen_context (synchronous read) | Cross-context PREPARING truth read to close the cancel-vs-advance race |
| concurrent writers of one OrderEntity row | Two cancel requests, or cancel vs confirmation projection, racing the same order |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-02-01 | Tampering / Info Disclosure | customer cancel entry point (IDOR) | mitigate | Resolve via `orderRepository.findByIdAndUserId(orderId, userId)`; empty → `orderNotFound()` (404, does not confirm existence) — existing project idiom |
| T-18-02-02 | Elevation of Privilege | window/state guard bypass | mitigate | `EnumSet.of(SUBMITTED,PENDING_CONFIRMATION,CONFIRMED)` guard after acquiring `lockById` PESSIMISTIC_WRITE; reject with `cancelWindowClosed()` |
| T-18-02-03 | Tampering (TOCTOU) | cancel vs kitchen QUEUED→PREPARING race | mitigate | Synchronous `KitchenItemStatusPort.findStatuses` read inside the locked tx; exclude/reject lines at/after PREPARING. Residual small window (Kafka at-least-once) documented; Plan 06 kitchen-void is the defensive backstop |
| T-18-02-04 | Repudiation / Tampering | duplicate/partial event emission | mitigate | Single `OutboxWriter.save` (MANDATORY propagation) in the same commit as the status/total change — no event without a committed cancel, no cancel without an event |
| T-18-02-SC | Tampering | dependency installs | accept | No new dependencies |
</threat_model>
