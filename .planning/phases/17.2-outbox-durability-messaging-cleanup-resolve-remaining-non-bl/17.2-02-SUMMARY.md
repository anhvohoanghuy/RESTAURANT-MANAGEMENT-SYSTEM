# Phase 17.2 Plan 02: Relay Per-Row Transaction Summary

**Split OutboxRelay's whole-batch `@Transactional poll()` into a per-row `OutboxRowPublisher.publish()` bean so a mid-batch Kafka send failure can no longer roll back sibling rows already marked SENT.**

## Performance

- **Duration:** ~20 min
- **Started:** 2026-07-10T02:22:00Z (approx.)
- **Completed:** 2026-07-10T02:42:42Z
- **Tasks:** 2 (TDD: RED then GREEN)
- **Files modified:** 3 (1 created, 2 modified)

## Accomplishments
- New `OutboxRowPublisher` (`@Component`, `@Transactional publish(OutboxEventEntity row)`) performs send-then-mark for exactly one claimed row and saves it, committing independently through the Spring proxy.
- `OutboxRelay.poll()` de-`@Transactional`-ed: it now only claims the batch and delegates each row to `outboxRowPublisher.publish(row)` through the injected bean (never self-invocation), matching the plan's `rowPublisher\.publish` key-link pattern.
- New test `siblingRowsAreNotRolledBackWhenMiddleRowFails` proves a three-row batch `[ok, fail, ok]` ends with both `ok` rows `SENT` (with `sentAt` set, `attempts == 0`) and the failing row left `PENDING` with `attempts == 1`, and structurally verifies `OutboxRowPublisher.publish` is invoked once per row.
- Claim semantics (`FOR UPDATE SKIP LOCKED`), `attempts`/`MAX_ATTEMPTS`→`FAILED` transition, `SEND_TIMEOUT_SECONDS`, and all existing log messages preserved unchanged (moved to `OutboxRowPublisher`).
- Send-then-mark ordering preserved: a row is never flipped to `SENT` before `KafkaTemplate.send(...).get()` returns successfully — the CR-01-adjacent durability guarantee from Phase 17.1 is intact.
- Full Maven suite green: 208 tests, 0 failures, 0 errors, 0 skipped.

## Decisions Made
- Explicit `outboxEventRepository.save(row)` added at the end of `OutboxRowPublisher.publish()` (not present in the old `OutboxRelay.publish()`, which relied on JPA dirty-checking within the single enclosing `@Transactional poll()`). Because `poll()` is no longer transactional, the row returned by `claimPending` is not guaranteed to remain a managed entity across the call into a *different* bean's new transaction, so an explicit save makes the per-row commit correct in production without relying on incidental managed-entity behavior. This is a Rule 2 addition (missing-but-necessary correctness step for the new architecture), not scope creep — it directly serves the plan's "per-row commit" artifact contract (`@Transactional`, `contains: "@Transactional"`) and does not touch the ledger/idempotency pattern.
- Kept `BATCH_SIZE` in `OutboxRelay` (claim-side) and moved `MAX_ATTEMPTS`/`SEND_TIMEOUT_SECONDS` into `OutboxRowPublisher` (publish-side), matching the plan's constant-ownership split by responsibility.
- Test verification of `OutboxWriterPersistenceTest,OutboxWireFormatRoundTripTest` per the plan's acceptance criteria: `OutboxWireFormatRoundTripTest` does not exist yet in this worktree (expected per the plan's own caveat — it is plan 01's deliverable, which may run in parallel/before this plan lands), so only `OutboxRelayTest,OutboxWriterPersistenceTest` were run together; both pass. The full Maven suite (208 tests) was also run as a final safety net and is green.
