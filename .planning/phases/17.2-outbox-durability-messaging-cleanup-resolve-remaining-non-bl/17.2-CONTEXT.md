<decisions>
## Implementation Decisions

### WR-01 — Outbox wire-format round-trip (highest priority)
- **Problem:** `OutboxWriter` serializes events with the Spring-injected `tools.jackson.databind.ObjectMapper`; the relay republishes that JSON verbatim via `StringSerializer`; consumers deserialize with `JacksonJsonDeserializer` (its own mapper). Nothing guarantees the two mappers agree on `Instant` and `BigDecimal` encoding. The relay is disabled in tests, so no test exercises the real serialize→publish→deserialize path.
- **Locked decision:** Add an integration test that round-trips a representative saga event containing an `Instant` and a `BigDecimal` through the ACTUAL OutboxWriter serialization → the ACTUAL consumer `JacksonJsonDeserializer` config, asserting the deserialized object equals the original. If a mismatch is found, align the two mappers (prefer: make `OutboxWriter` use the same ObjectMapper/`JavaTimeModule` + `WRITE_DATES_AS_TIMESTAMPS`/`WRITE_BIGDECIMAL_AS_PLAIN` settings the consumer deserializer uses). Do NOT change the on-wire format silently — if a format change is required, confirm both producer and consumer agree.
- **Acceptance:** a failing-then-passing test proves `Instant`+`BigDecimal` survive the OutboxWriter→JacksonJsonDeserializer round-trip byte-for-byte-compatibly.

### WR-02 — Relay per-row transaction (avoid duplicate-publish amplification)
- **Problem:** `OutboxRelay.poll()` is `@Transactional` and awaits up to 100 sends (each up to 10s) inside ONE transaction while holding `FOR UPDATE SKIP LOCKED` row locks. A rollback of that long transaction after some rows were already sent to Kafka re-publishes the whole batch on the next poll.
- **Locked decision:** Publish/flip-to-SENT each claimed row in its OWN transaction (per-row commit) rather than one transaction for the whole batch. Keep the send-then-mark-SENT ordering (never mark SENT before broker ack — preserves the CR-01-adjacent durability guarantee). Preserve claim semantics (`FOR UPDATE SKIP LOCKED`) and the `attempts`/`MAX_ATTEMPTS`→`FAILED` behavior. Keep the batch claim, but commit per row so a slow/failed send cannot roll back already-acknowledged sends.
- **Acceptance:** a test proves that a send failure on row N does not roll back / re-drive rows already marked SENT in the same poll batch.

### WR-03 — Delete dead publishers
- **Problem:** After the outbox cutover, `KafkaOrderEventPublisher` and `KafkaInventoryStockResultPublisher` have zero callers in `src/main` but still carry whenComplete logging — misleading dead code implying a live direct-publish path.
- **Locked decision:** Delete both publisher classes AND any now-unused Kafka producer beans/config wired only for them. Verify by grep that nothing in `src/main` references them before deletion; update/remove any tests that only exercised them. Do NOT delete producer config still used by the outbox relay (`OutboxConfig`) or by other live producers (Payment/Table).
- **Acceptance:** grep of `src/main` returns no reference to either deleted class; suite stays green.

### IN-02 — Outbox retention + FAILED alerting
- **Problem:** `outbox_events` rows in `SENT` state are never pruned (unbounded table growth → slowing `claimPending`); `FAILED` rows die silently with only a `log.error`.
- **Locked decision:** Add a scheduled retention job that deletes `SENT` rows older than a configurable retention window (default e.g. 7 days, property-driven, gated off in tests like the relay). Add a lightweight surfacing of `FAILED` rows (at minimum a scheduled count + `log.warn`/metric so a poison message is visible to operators; a full alerting integration is out of scope). Reuse the existing `@ConditionalOnProperty`/scheduling pattern from `OutboxRelay`/`OutboxConfig`.
- **Acceptance:** a test proves SENT rows past the retention cutoff are deleted and rows within the window are retained; FAILED-row surfacing has a unit test.

### IN-03 — Unify reason-length constant
- **Problem:** `OrderConfirmationService.MAX_REASON_LEN = 60000` and `OrderEntity.rejectionReason @Column(length = 65535)` are unaligned magic numbers.
- **Locked decision:** Define one shared source of truth for the rejection-reason cap and derive both the truncation cap and (where feasible) the column length from it, or document the relationship with a single named constant referenced by both. Keep behavior identical (truncation still caps at the same effective length).
- **Acceptance:** there is a single named constant; changing it in one place is reflected consistently; existing truncation test still passes.

### WR-05 — Explicit fulfillment rank map (remove enum-ordinal dependency)
- **Problem:** `KitchenStatusProjectionService.deriveTargetStatus` compares `KitchenItemStatus.ordinal()` values; reordering the enum silently breaks the derivation.
- **Locked decision:** Replace ordinal comparisons with an explicit rank map for `KitchenItemStatus` (mirroring the existing `FULFILLMENT_RANK` map for `OrderStatus`). Behavior must be identical to today. Add/keep tests that would catch a future enum reorder.
- **Acceptance:** no `.ordinal()` comparison remains in the derivation; a test pins the intended ordering independent of enum declaration order.

### WR-06 — Externalize DB credentials
- **Problem:** `application.properties` hard-codes `spring.datasource.username=root` / `password=123456` in plaintext.
- **Locked decision:** Move DB username/password to environment-variable placeholders with safe local-dev defaults, e.g. `${DB_USERNAME:root}` / `${DB_PASSWORD:123456}` (matching the existing `${AUTH_...}` idiom already in the file). Update any developer/README docs and `.env.example` if such a file exists. Do NOT commit real secrets. Confirm the app still starts with the placeholders resolved.
- **Acceptance:** `application.properties` contains no bare plaintext DB password; the value is env-driven with a documented local default.

### Claude's Discretion
- Exact retention window default, property names, scheduling intervals, and whether FAILED-surfacing uses a metric vs log — pick sensible defaults consistent with existing outbox config idioms.
- Plan/wave decomposition and which findings can run in parallel (WR-03/IN-03/WR-05/WR-06 are largely independent; WR-01/WR-02/IN-02 all touch the outbox subsystem and may need care to avoid file overlap).
