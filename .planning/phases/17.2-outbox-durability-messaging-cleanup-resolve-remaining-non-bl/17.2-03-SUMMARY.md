# Phase 17.2 Plan 03: Outbox SENT-row retention + FAILED-row surfacing Summary

**Scheduled, property-gated `OutboxRetentionJob` deletes SENT outbox rows past a configurable TTL (default 7d/hourly) via a bulk JPQL delete filtered strictly on status, and log.warns a FAILED-row count each hour so poison messages are no longer silent — off by default in tests via `outbox.retention.enabled=false`.**

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Bulk JPQL delete left a stale entity visible via findById in the same transaction**
- **Found during:** Task 2 (writing the real-DB retention test)
- **Issue:** The `@Modifying @Query("DELETE FROM ... WHERE status = :status AND createdAt < :cutoff")` query executed correctly against the DB (the returned delete count was correct), but a subsequent `findById` on the same still-open Spring-test transaction returned the just-deleted row anyway — bulk JPQL DELETE/UPDATE bypasses the Hibernate first-level (persistence-context) cache, so the previously-loaded managed entity remained visible to reads in the same transaction/session even though the underlying DB row was gone.
- **Fix:** Added `clearAutomatically = true, flushAutomatically = true` to the `@Modifying` annotation, so the persistence context is flushed before the delete runs and cleared after, making subsequent reads in the same transaction hit the DB.
- **Files modified:** `src/main/java/com/example/feat1/DDD/shared/outbox/repository/OutboxEventRepository.java`
- **Verification:** `OutboxRetentionJobTest.deletesOnlyOldSentRowsAndRetainsWithinWindowPendingAndFailedRows` passes; full Maven suite green
- **Committed in:** `cfaebc5` (part of Task 2 commit)

**2. [Rule 4 boundary — resolved without architectural change] No `@DataJpaTest` slice module available**
- **Found during:** Task 2 (test authoring, per plan's suggested `@DataJpaTest` approach)
- **Issue:** The plan's `<action>` suggested `@DataJpaTest` for the retention-delete test. This Spring Boot 4.0.6 dependency setup has no dedicated JPA test-slice artifact on the classpath (`spring-boot-test-autoconfigure` is present, but `DataJpaTest` itself was not resolvable — compile failed with `package org.springframework.boot.test.autoconfigure.orm.jpa does not exist`).
- **Resolution (not an architectural change — no new dependency, no scope change):** Used the codebase's existing, already-proven `@SpringBootTest @Transactional` + `@Autowired OutboxEventRepository` pattern (identical to `OutboxWriterPersistenceTest`) against the real H2 test datasource instead. This satisfies the plan's "and/or ... assert only old SENT rows are gone" acceptance criterion via the same real-database mechanism the plan intended, just via the pattern already established in this codebase rather than a slice annotation unavailable here.
- **Files modified:** `src/test/java/com/example/feat1/DDD/shared/outbox/OutboxRetentionJobTest.java`
- **Verification:** Test compiles and passes; no new Maven dependency added
- **Committed in:** `cfaebc5` (Task 2 commit)
