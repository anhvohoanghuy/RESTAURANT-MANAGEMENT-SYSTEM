<objective>
Close WR-02: `OutboxRelay.poll()` is `@Transactional` and awaits up to 100 sends (each up to 10s)
inside ONE transaction while holding `FOR UPDATE SKIP LOCKED` row locks. If that long transaction
rolls back after some rows were already sent to Kafka, the whole batch re-publishes on the next poll
(duplicate-publish amplification). Commit each claimed row in its OWN transaction so a slow/failed
send on one row cannot roll back sibling rows already acknowledged and marked SENT.

Purpose: Bound duplicate publishing to at-most the single in-flight row on failure, instead of the
entire in-flight batch.
Output: A per-row transactional publish unit, a de-`@Transactional`-ed `poll()`, and a test proving
sibling-row isolation on failure.
</objective>
