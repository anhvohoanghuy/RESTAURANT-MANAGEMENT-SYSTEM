# Phase 17.2: Code Review Report

**Reviewed:** 2026-07-10
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Phase 17.2 resolves deferred 17.1 findings: the outbox relay was split into a de-transactionalized `poll()` plus a per-row `@Transactional OutboxRowPublisher.publish()` (WR-02), a scheduled `OutboxRetentionJob` was added (IN-02), the rejection-reason cap was unified into `OrderEntity.REJECTION_REASON_MAX_LEN` (IN-03), enum-ordinal comparisons were replaced by an explicit `ITEM_RANK` map (WR-05), and DB credentials were externalized (WR-06).

**CR-01 non-regression: CONFIRMED.** In both `KitchenStatusProjectionService.onTicketStatusChanged` (line 130) and `OrderConfirmationService.onStockResult` (line 96), `recordProcessed(...)` remains the LAST statement in the single `@Transactional` handler, with no `REQUIRES_NEW` pre-commit ledger insert. The idempotency ledger commits atomically with the business write. The CR-01 fix is intact. `OutboxRowPublisher` also correctly avoids reintroducing a `REQUIRES_NEW` pattern.

The retention bulk-delete is correctly scoped (`status='SENT' AND createdAt < cutoff`, `@Modifying(clearAutomatically, flushAutomatically)`) and cannot over-delete PENDING/FAILED rows. `ITEM_RANK` covers all five `KitchenItemStatus` values, so `itemRank()` cannot NPE for valid enum inputs, and `attempts` is a primitive `int` so the increment is null-safe.

The dominant concern is the new per-row transaction boundary: de-transactionalizing `poll()` releases the `FOR UPDATE SKIP LOCKED` row locks *before* `publish()` re-opens each row, which silently defeats the cross-instance de-duplication that `claimPending` was written to provide (WR-01). Additional warnings cover incomplete per-row isolation, a VARCHAR/TEXT mapping mismatch on the widened column, a committed weak DB-password default, and swallowed `InterruptedException`.

## Warnings

### WR-01: De-transactionalized `poll()` releases SKIP LOCKED locks before publish → cross-instance double-publish

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRelay.java:37-43`, `src/main/java/com/example/feat1/DDD/shared/outbox/repository/OutboxEventRepository.java:20-30`

**Issue:** `poll()` is now (correctly, per WR-02) NOT `@Transactional`. But `claimPending` is a Spring Data custom `@Query` method, which is not wrapped in a transaction by default. Its `SELECT ... FOR UPDATE SKIP LOCKED` therefore executes in its own short-lived unit of work, and the acquired row locks are **released the moment `claimPending` returns** — before `outboxRowPublisher.publish(row)` opens its own per-row transaction. `claimPending` does not transition rows to any intermediate "claimed"/"processing" status either; the returned rows stay `status='PENDING'` until `publish()` flips them.

Consequently the ONLY protection `SKIP LOCKED` provided — preventing a second app instance from claiming the same rows (its own Javadoc: "lets concurrent app instances poll the same table without double-publishing a row") — no longer holds. A concurrent instance's `poll()` will re-`claimPending` the still-PENDING, now-unlocked rows and publish them again. Under the previous `@Transactional poll()`, the locks were held for the whole batch, which is exactly what this refactor removed.

Impact is bounded (the outbox is at-least-once and downstream consumers are idempotent via the `order_processed_events` ledger), so this is duplicate amplification rather than data loss — but the documented multi-instance guarantee is now false. Single-instance deployments are unaffected because `@Scheduled(fixedDelay)` serializes `poll()`.

**Fix:** Either (a) make the claim durable by transitioning rows to an intermediate status inside a short claim transaction (e.g. `UPDATE ... SET status='CLAIMING' WHERE id IN (...)`) so publish operates on state the next poller will skip, or (b) explicitly document and accept that cross-instance dedup is no longer provided and update the `claimPending`/`OutboxRelay` Javadoc to stop asserting multi-instance safety. Do not rely on `FOR UPDATE SKIP LOCKED` spanning `poll()` → `publish()`; it does not.

### WR-02: `poll()` does not isolate each `publish()` call — one unexpected exception aborts the rest of the batch

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRelay.java:40-42`, `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRowPublisher.java:75`

**Issue:** The stated goal of WR-02 is that "a slow or failed send on one claimed row can no longer roll back sibling rows." `publish()` catches send failures internally, but the final `outboxEventRepository.save(row)` (line 75) is OUTSIDE the try/catch. If `save()` throws (constraint/DB error, or a merge failure), the exception propagates out of `publish()`, rolls back that row's transaction, and then propagates into `poll()`'s `for` loop — which has no try/catch — aborting publication of every remaining claimed row this cycle. Per-row isolation is therefore incomplete against persistence-layer failures.

**Fix:** Wrap the per-row call in `poll()` so one row's failure cannot starve its siblings:
```java
for (OutboxEventEntity row : claimed) {
  try {
    outboxRowPublisher.publish(row);
  } catch (Exception ex) {
    log.warn("Outbox row {} publish transaction failed; continuing batch", row.getId(), ex);
  }
}
```

### WR-03: `rejection_reason` maps to `VARCHAR(60000)`, not TEXT as the Javadoc claims; risks MySQL DDL failure

**File:** `src/main/java/com/example/feat1/DDD/order_context/infrastructure/entity/OrderEntity.java:48`, `src/main/java/com/example/feat1/DDD/order_context/application/OrderConfirmationService.java:150-153`

**Issue:** `@Column(name="rejection_reason", length = REJECTION_REASON_MAX_LEN)` with no `columnDefinition`/`@Lob`. With Hibernate's MySQL dialect (`maxVarcharLength = 65535`), a length of 60000 maps to `VARCHAR(60000)`, not TEXT. On MySQL with the default `utf8mb4` charset (no charset override in the datasource URL), the maximum declarable `VARCHAR` is 16383 characters; `VARCHAR(60000)` is rejected at DDL time with "Column length too big (max = 16383); use BLOB or TEXT instead," which can break `ddl-auto=update` on a fresh schema. Meanwhile `OrderConfirmationService.truncate` Javadoc asserts "the column is widened to TEXT" — a documentation/implementation mismatch. Note the sibling long-text column (`OutboxEventEntity.payload`) correctly uses `@Lob` for exactly this reason. (This is charset/existing-schema dependent — hence WARNING not BLOCKER — and the prior value 65535 had the same latent issue.)

**Fix:** Make the mapping match the stated intent, mirroring `OutboxEventEntity.payload`:
```java
@Lob
@Column(name = "rejection_reason")
private String rejectionReason;
```
(or `@Column(name = "rejection_reason", columnDefinition = "TEXT")`). The app-level char-count truncation at `MAX_REASON_LEN` remains a valid defensive bound regardless.

### WR-04: Externalized DB password keeps a real weak default (`123456`) committed in source

**File:** `src/main/resources/application.properties:4`

**Issue:** WR-06 externalized the credential as `spring.datasource.password=${DB_PASSWORD:123456}`, but the fallback default is a real, usable password baked into the repository. If a production deployment forgets to set `DB_PASSWORD`, the app silently starts with `123456` (fail-open) rather than refusing to boot. This partially undoes the intent of externalizing the secret.

**Fix:** Drop the default so an unset variable fails fast (`${DB_PASSWORD}`), or use an obviously-invalid placeholder that cannot match any real DB (as done for `jwt.secret`'s `change-me-...`). Keep local-dev convenience in `.env.example`, not in the committed property default.

### WR-05: `InterruptedException` swallowed in `publish()` without restoring the interrupt flag

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRowPublisher.java:47-52`

**Issue:** `future.get(SEND_TIMEOUT_SECONDS, ...)` can throw `InterruptedException`, which is absorbed by the broad `catch (Exception ex)`. The catch treats it as a normal retryable send failure and never calls `Thread.currentThread().interrupt()`, so the thread's interrupt status is cleared and swallowed. During shutdown or thread-pool reclamation this can mask a cancellation signal and delay a clean stop.

**Fix:** Handle interruption explicitly and restore the flag:
```java
} catch (InterruptedException ie) {
  Thread.currentThread().interrupt();
  row.setAttempts(row.getAttempts() + 1);
  log.warn("Outbox row {} publish interrupted", row.getId(), ie);
} catch (Exception ex) {
  // existing attempts/FAILED handling
}
```

## Info

### IN-01: Negative/zero `outbox.retention.ttl-days` would delete all SENT rows

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRetentionJob.java:43-51`

**Issue:** `ttlDays` is read from config with no lower-bound validation. A misconfigured negative value makes `cutoff` a future instant, so `deleteByStatusAndCreatedAtBefore("SENT", cutoff)` deletes every SENT row (still bounded to SENT, so no PENDING/FAILED loss, but it defeats retention). A value of 0 deletes SENT rows created up to "now."

**Fix:** Guard at startup or in the sweep, e.g. `if (ttlDays < 1) { log.warn(...); return; }`, or clamp to a sane minimum.

### IN-02: `deriveTargetStatus` recomputes item ranks with repeated full-stream passes

**File:** `src/main/java/com/example/feat1/DDD/order_context/application/KitchenStatusProjectionService.java:141-161`

**Issue:** Each tier check re-streams `items` and re-invokes `itemRank()`/`ITEM_RANK.get()` per element. Logic is correct and readable; this is a minor clarity/maintainability note only (performance is out of v1 scope). The four sequential `allMatch`/`anyMatch` passes could be a single reduce over the minimum item rank if this hot path ever grows, but no change is required now.

**Fix:** Optional — compute `int minRank = items.stream().mapToInt(i -> itemRank(i.status())).min().orElse(-1);` once and branch on it. Advisory only.
