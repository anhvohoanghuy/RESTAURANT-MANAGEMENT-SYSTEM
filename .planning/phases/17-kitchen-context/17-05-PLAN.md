  <behavior>
    - Valid single-step advance (e.g. QUEUED→PREPARING, PREPARING→READY): status persisted; returns KitchenItemResponse with the new status
    - Illegal transition (skip QUEUED→READY, revert READY→PREPARING, advance from terminal COMPLETED): throws KitchenDomainException.transitionInvalid(), status unchanged, no events published
    - QUEUED→PREPARING: publishes SettleTriggerEvent exactly once with orderId, item.orderLineId, totalLines = ticket.getItems().size(), AND publishes KitchenTicketStatusChangedEvent
    - Any other valid transition (PREPARING→READY, etc.): publishes KitchenTicketStatusChangedEvent but NOT a SettleTriggerEvent
    - Item not found for the given (orderId, itemId): throws KitchenDomainException.itemNotFound() (IDOR: item of a different order is not found)
    - Both publishes are registered after-commit (registerSynchronization), never mid-transaction
  </behavior>
  <action>
    `KitchenTicketAdvanceService` (`@Service @RequiredArgsConstructor`). Method
    `@Transactional public KitchenItemResponse advance(UUID orderId, UUID itemId, AdvanceItemStatusRequest
    request, UUID actorId)`: FIRST call `itemRepository.lockByOrderIdAndItemId(orderId, itemId)` (dual-key
    PESSIMISTIC_WRITE) → `.orElseThrow(KitchenDomainException::itemNotFound)` — locking before the check
    closes Pitfall 1. Read `current = item.getStatus()`, `target = request.targetStatus()`. Guard with a
    private `isValidTransition(KitchenItemStatus current, KitchenItemStatus target)` `switch` allowing ONLY
    the immediate next state per case (`QUEUED→PREPARING`, `PREPARING→READY`, `READY→SERVED`,
    `SERVED→COMPLETED`, `COMPLETED→false`); on false throw `KitchenDomainException.transitionInvalid()`
    with NO mutation. On valid: `item.setStatus(target)`, save. Then register after-commit publishes using
    a `publishAfterCommit` helper copied from OrderSubmissionService: (a) IF `current == QUEUED && target ==
    PREPARING`, publish `new SettleTriggerEvent(UUID.randomUUID(), SettleTriggerEvent.TYPE, Instant.now(),
    orderId, item.getOrderLineId(), item.getTicket().getItems().size())` via `KitchenSettleTriggerPublisher`;
    (b) ALWAYS publish a `KitchenTicketStatusChangedEvent` built from the ticket's current items snapshot
    (map each item to `ItemStatus(orderLineId, status)`) via `KitchenTicketStatusChangedPublisher`. Return
    a mapped `KitchenItemResponse`. Test with mocked repository + both publisher ports covering every
    behavior above; for exactly-once assert `verify(settleTriggerPublisher, times(1))` on QUEUED→PREPARING
    and `times(0)` on other transitions; for the concurrent-double-advance case, simulate the second
    call reading the now-PREPARING row (lock returns the mutated item) and assert it rejects as illegal so
    the trigger publishes only once.

| Boundary | Description |
|----------|-------------|
| concurrent PATCH → same item | Two transactions must serialize on the item row |
| URL `{orderId}`/`{itemId}` → item | Item must belong to the path's order |
| advance transaction → Kafka | Events must reflect only committed transitions |
