# Plan 06-01: Implement Auth Hardening

<objective>
Add auth hardening across rate limiting, local lockout, audit logging, refresh-session metadata, and self-service session management while preserving existing auth response contracts.
</objective>

<tasks>

1. Add GSD requirement and roadmap artifacts for Phase 06.
2. Implement Redis-backed `AuthRateLimitService` and `LoginLockoutService`.
3. Add persistent security audit event model, repository, and service.
4. Extend refresh-token persistence with IP, user-agent, and last-used metadata.
5. Add `AuthSessionService` and `/auth/sessions` endpoints for listing and revocation.
6. Wire hardening into register, login, Google login, refresh, logout, email verification, and password reset flows.
7. Add focused unit and integration tests.
8. Run `mvnw.cmd test`.

</tasks>

<verification>

- Unit tests cover limiter, lockout, audit, and session ownership.
- Integration tests cover lockout, rate-limit error contract, audit creation, session listing, revoke-one, and revoke-others behavior.
- Full Maven test suite passes.

</verification>

<success_criteria>

- Existing auth contracts remain unchanged.
- New hardening errors use stable codes.
- Session responses never include refresh-token values.
- Redis failure is fail-open for rate/lock checks.
- Audit persistence is best effort for normal auth flows.

</success_criteria>
