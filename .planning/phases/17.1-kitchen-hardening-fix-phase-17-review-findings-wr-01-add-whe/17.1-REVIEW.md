# Phase 17.1: Code Review Report

**Reviewed:** 2026-07-09T04:05:31Z
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

Phase 17.1 hardened the kitchen/order/inventory Kafka messaging: `whenComplete` send-failure
logging, a shared transactional outbox (`OutboxWriter` / `OutboxRelay` / `OutboxEventEntity`),
`REQUIRES_NEW` idempotency-ledger writers, fail-closed rank projection, and an advance audit trail.

The outbox write-in-transaction + relay-poll mechanism is structurally sound: the payload is
serialized and persisted in the caller's transaction (`MANDATORY` propagation), and the relay flips
`PENDING -> SENT` only after the broker acknowledges the send. The fail-closed rank guard in
`KitchenStatusProjectionService` (K-WR-03) is correct — an unknown current rank is now skipped
rather than fail-open.

However, the review found one BLOCKER: the `REQUIRES_NEW` idempotency-ledger pattern adopted across
all four event consumers commits the "processed" marker in a **separate** transaction that is
**not** rolled back when the business work fails. This trades the spurious-rollback problem it was
meant to fix for a strictly worse silent-event-loss problem — a failure in the narrow window after
the ledger commits leaves the aggregate permanently stuck with no recovery path. Additional warnings
cover unverified outbox wire-format compatibility, batch-transaction failure amplification in the
relay, dead publisher code left with newly-added logging, and a residual dual-write gap in the
kitchen advance path that the outbox was supposed to close.

No structural pre-pass (`<structural_findings>`) was provided. All reviewed files are Java, so the
JS/TS CONVENTION rule packs skip (no CONVENTION findings apply).

## Critical Issues

### CR-01: REQUIRES_NEW idempotency ledger commits "processed" before the business work commits — a post-ledger failure silently loses the event and strands the aggregate

**File:** `src/main/java/com/example/feat1/DDD/order_context/application/OrderConfirmationService.java:53-89`
(same defect in
`src/main/java/com/example/feat1/DDD/inventory_context/application/InventoryReservationService.java:71-160`,
`src/main/java/com/example/feat1/DDD/order_context/application/KitchenStatusProjectionService.java:61-111`,
`src/main/java/com/example/feat1/DDD/kitchen_context/application/KitchenTicketCreationService.java:41-74`;
ledger writers `OrderLedgerWriter.java:41-59`, `KitchenLedgerWriter.java:41-59`,
`InventoryLedgerWriter` are all `@Transactional(propagation = REQUIRES_NEW)`)

**Issue:** Every consumer follows the same sequence inside its outer `@Transactional` business
method:

1. `existsByEventIdAndConsumerName(...)` fast pre-check (false on first delivery).
2. `ledgerWriter.tryInsert(eventId, consumer)` — runs in `REQUIRES_NEW`, `saveAndFlush` + returns,
   so the ledger row **commits immediately in its own inner transaction**.
3. Business mutation + `outboxWriter.save(...)` in the outer transaction.
4. Outer transaction commits.

If anything between step 2's commit and step 4's commit fails — outer-transaction commit failure
(deadlock, connection reset, constraint), an exception in `outboxWriter.save` /
`ObjectMapper.writeValueAsString`, or (see IN-01) an NPE while building the event — the **outer
transaction rolls back but the ledger row from step 2 stays committed.** On Kafka redelivery, the
step-1 pre-check now returns `true`, so the handler returns normally (a silent no-op success). The
`DefaultErrorHandler` retry/DLT safety net never engages because the retried delivery no longer
throws. Concrete consequence per consumer:

- `OrderConfirmationService`: order stays `PENDING_CONFIRMATION` forever — never `CONFIRMED`, never
  `REJECTED`, no `OrderConfirmed` outbox row, kitchen never gets a ticket.
- `InventoryReservationService`: no reservation, no `OrderStockResult` published — the order's saga
  never receives a verdict and stalls in `PENDING_CONFIRMATION`.
- `KitchenTicketCreationService`: no kitchen ticket is ever created for that order.
- `KitchenStatusProjectionService`: the fulfillment snapshot is dropped and the order status lags.

The `REQUIRES_NEW` isolation was introduced (WR-01) to stop a concurrent-duplicate
`DataIntegrityViolationException` from marking the outer transaction rollback-only. But in the
concurrent-duplicate case the losing thread *should* roll back — it is a duplicate. By committing
the ledger marker independently and ahead of the work, the design converts a safe, self-healing
rollback (Kafka redelivers, reprocesses cleanly) into unrecoverable event loss. The inbox/dedup
insert must be atomic with the work it guards.

**Fix:** Insert the idempotency-ledger row in the **same** transaction as the business mutation so
they commit or roll back together, and treat the unique-constraint violation as the dedup signal.
For example, drop the `REQUIRES_NEW` writer and do the insert inline:

```java
@Transactional
public void onStockResult(OrderStockResultEvent event) {
  if (processedEventRepository.existsByEventIdAndConsumerName(event.eventId(), CONSUMER_NAME)) {
    return; // fast path for already-processed
  }
  // ... load order, apply verdict, outboxWriter.save(...) ...

  // Same-transaction dedup insert LAST: a concurrent duplicate throws here and rolls back
  // this whole transaction (including the business mutation) — which is correct, the winner
  // already applied it. A genuine first delivery commits ledger + work + outbox atomically.
  try {
    processedEventRepository.saveAndFlush(newLedgerRow(event.eventId(), CONSUMER_NAME));
  } catch (DataIntegrityViolationException duplicate) {
    throw new ConcurrentDuplicateException(duplicate); // let the transaction roll back
  }
}
```

If the outer transaction must survive the duplicate without a rollback for throughput reasons, an
acceptable alternative is to keep `REQUIRES_NEW` but insert the ledger row as the **last** step
after the business commit is guaranteed — never before it. The current ordering (mark-processed
first, do-work-second, no compensation) is the defect.

## Warnings

### WR-01: Outbox wire-format compatibility with consumers is asserted but never verified (Instant / BigDecimal)

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/application/OutboxWriter.java:40`,
`src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxConfig.java:24-25`

**Issue:** The outbox serializes events with the injected `tools.jackson.databind.ObjectMapper`
(Spring-managed) and republishes the resulting JSON verbatim via `StringSerializer`. The retired
direct path (`KafkaOrderEventPublisher`, `KafkaInventoryStockResultPublisher`) used Spring Kafka's
`JacksonJsonSerializer`, which uses its **own** internally-configured `ObjectMapper`, not the
Spring-managed bean. `OutboxConfig`'s Javadoc claims the two produce a "byte-identical wire
payload," but that equality is unverified and depends on both mappers agreeing on `Instant`
(ISO-8601 string vs epoch number, `WRITE_DATES_AS_TIMESTAMPS`) and `BigDecimal` (plain vs scientific
notation) formatting. The events in scope (`OrderCreatedEvent`, `OrderConfirmedEvent`,
`OrderStockResultEvent`) all carry `Instant` and `BigDecimal`. If the mappers diverge, every
outbox-published message fails to deserialize at the consumer and lands on the DLT.

**Fix:** Add an integration test that serializes a representative event through `OutboxWriter`'s
`ObjectMapper`, publishes the stored payload as a `String`, and asserts the consumer's
`JacksonJsonDeserializer` reconstructs it equal to the original — covering `Instant` and
`BigDecimal` fields. Alternatively, explicitly inject the same `ObjectMapper` instance into both the
`OutboxWriter` and the Kafka `JacksonJsonSerializer`/`JacksonJsonDeserializer` so the equivalence is
enforced by construction rather than by assumption.

### WR-02: OutboxRelay awaits every send inside one batch transaction — a slow broker can time out the transaction and roll back already-committed SENT flags, re-publishing the whole batch

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRelay.java:45-84`

**Issue:** `poll()` claims up to `BATCH_SIZE = 100` rows in one `@Transactional` and, for each,
awaits the send synchronously with `.get(SEND_TIMEOUT_SECONDS = 10, SECONDS)`. All `SENT`/`attempts`
mutations are managed-entity changes flushed only at the single end-of-`poll` commit. If the broker
is slow/unreachable, a batch can hold the transaction (and the `FOR UPDATE SKIP LOCKED` row locks +
a DB connection) for up to ~100 × 10s. If the surrounding transaction/connection timeout fires
before `poll` returns, the commit is lost and **every** status flip in the batch rolls back —
including rows that were already successfully sent to the broker — so the next poll re-publishes
them. Consumers are idempotent (at-least-once is the contract), but this amplifies duplicates well
beyond the single-message crash window the outbox targets.

**Fix:** Bound the blast radius: commit per row (or in small chunks) rather than one transaction per
100-row batch — e.g., process each claimed row in its own short transaction, or reduce `BATCH_SIZE`
and enforce a producer `delivery.timeout.ms` / `max.block.ms` that guarantees the total batch stays
well under the transaction timeout. Persisting each `SENT` flip promptly ensures a mid-batch failure
never rolls back an already-published row.

### WR-03: Dead publisher beans retained — `whenComplete` logging was added to code paths that are no longer invoked

**File:** `src/main/java/com/example/feat1/DDD/order_context/infrastructure/adapter/KafkaOrderEventPublisher.java`,
`src/main/java/com/example/feat1/DDD/inventory_context/infrastructure/adapter/KafkaInventoryStockResultPublisher.java`

**Issue:** `OrderSubmissionService` and `OrderConfirmationService` now publish via `OutboxWriter`,
and `InventoryReservationService` writes its result to the outbox. Grep confirms `publishOrderCreated`,
`publishOrderConfirmed`, and `publishStockResult` have **no callers** in `src/main` (only doc-comment
references). These publisher beans — and the `whenComplete` send-failure logging this phase added to
them — are now unreachable. Beyond being dead code, leaving both a live outbox path and a wired-but-
dormant direct-publish bean invites an accidental double-publish if either is re-injected later.

**Fix:** Delete `KafkaOrderEventPublisher` / `KafkaInventoryStockResultPublisher` (and their now-unused
`OrderEventPublisher` / `InventoryStockResultPublisher` ports and typed `KafkaTemplate` beans) if the
outbox is the sole publish path, or document explicitly why they are retained. Do not invest hardening
(logging) in code that no longer executes.

### WR-04: Kitchen advance still publishes via `afterCommit` (not the outbox) — the exact dual-write gap the outbox was built to close remains open here, and losing the settle trigger strands a reservation

**File:** `src/main/java/com/example/feat1/DDD/kitchen_context/application/KitchenTicketAdvanceService.java:59-137`

**Issue:** `advance` publishes `SettleTriggerEvent` and `KitchenTicketStatusChangedEvent` via
`TransactionSynchronization.afterCommit`. If the process crashes after the DB commit but before the
`afterCommit` callback runs, both events are lost with no retry — precisely the crash-between-commit-
and-send gap the transactional outbox closes for the order/inventory saga. The publisher's own log
message acknowledges the consequence: "reservation may never settle." Order and inventory were
migrated to the outbox in this phase; the highest-risk fulfillment path (per the class Javadoc) was
left on the weaker `afterCommit` mechanism, an inconsistent hardening result.

**Fix:** Route the kitchen `SettleTriggerEvent` (and, if delivery matters, the status-changed event)
through the same `OutboxWriter` in the advance transaction, consistent with order/inventory. If
keeping `afterCommit` is a deliberate scope decision, document the accepted settle-trigger loss risk
and the operational recovery procedure (e.g., a reconciliation sweep for reservations stuck `HELD`).

### WR-05: Fulfillment status derivation depends on `KitchenItemStatus` enum ordinal order — silent breakage if the enum is ever reordered

**File:** `src/main/java/com/example/feat1/DDD/order_context/application/KitchenStatusProjectionService.java:113-133`

**Issue:** `deriveTargetStatus` compares `item.status().ordinal() >= KitchenItemStatus.SERVED.ordinal()`
(and READY/PREPARING). This couples correctness to the declaration order of `KitchenItemStatus`.
Reordering or inserting an enum constant would silently change which order status is derived (e.g.,
an order could be reported `READY` while an item is only `PREPARING`) with no compile-time or test
signal. Enum ordinal is an implementation detail that should not carry business ordering semantics.

**Fix:** Give `KitchenItemStatus` an explicit numeric `rank()` (or an explicit ordered comparison
map, mirroring `FULFILLMENT_RANK` in the same file) and compare against that, so the ordering is
declared data rather than an accident of enum position.

### WR-06: Plaintext database credentials committed in application.properties

**File:** `src/main/resources/application.properties:3-4`

**Issue:** `spring.datasource.username=root` and `spring.datasource.password=123456` are hardcoded in
plaintext. This is inconsistent with the env-var-with-default pattern used for every other secret in
the same file (`jwt.secret=${JWT_SECRET:...}`, `google.oauth.client-ids=${GOOGLE_OAUTH_CLIENT_IDS:}`).
A committed credential leaks to anyone with repo access and cannot be rotated without a code change.

**Fix:** Externalize to an environment variable with a dev-only default, matching the existing
convention: `spring.datasource.username=${DB_USERNAME:root}` and
`spring.datasource.password=${DB_PASSWORD:}` (or a clearly-marked dev placeholder), sourcing the real
value from the environment in every non-local profile.

## Info

### IN-01: `KitchenTicketCreationService` iterates `event.lines()` and `line.selectedToppings()` with no null guard

**File:** `src/main/java/com/example/feat1/DDD/kitchen_context/application/KitchenTicketCreationService.java:69-85`

**Issue:** `for (OrderConfirmedLine line : event.lines())` and `for (... : line.selectedToppings())`
assume non-null collections. A malformed `OrderConfirmedEvent` (null lines or null toppings) throws
an NPE. Because the ledger row is already committed by that point (see CR-01), the resulting
rollback permanently drops the event. `InventoryReservationService.computeRequired` (lines 171-183)
correctly null-guards `event.lines()` and `line.selectedToppings()`; this consumer should match.

**Fix:** Null-guard both loops (skip or treat as empty), consistent with `computeRequired`.

### IN-02: SENT outbox rows are never pruned

**File:** `src/main/java/com/example/feat1/DDD/shared/outbox/infrastructure/OutboxRelay.java`,
`src/main/java/com/example/feat1/DDD/shared/outbox/entity/OutboxEventEntity.java`

**Issue:** Rows are flipped to `SENT` but never deleted or archived, so `outbox_events` grows
unbounded. The `ix_outbox_status_created` index keeps the `status='PENDING'` claim query selective,
so this is an operational/retention concern rather than a correctness one, but it will eventually
require intervention.

**Fix:** Add a scheduled purge (or partition/TTL) for `SENT` rows older than a retention window.

### IN-03: Rejection-reason cap (`MAX_REASON_LEN = 60000`) and column length (`65535`) are unaligned magic numbers

**File:** `src/main/java/com/example/feat1/DDD/order_context/application/OrderConfirmationService.java:41`,
`src/main/java/com/example/feat1/DDD/order_context/infrastructure/entity/OrderEntity.java:39`

**Issue:** The application caps `rejectionReason` at 60000 chars while the column is `length = 65535`.
The two bounds are defined independently with no shared constant, so they can silently drift apart;
the intent ("cap below the column limit") is only recoverable by reading both files.

**Fix:** Derive one from the other or add a comment tying them together (e.g., a shared constant, or
a note that 60000 is a deliberate headroom margin under the 65535 column limit).
