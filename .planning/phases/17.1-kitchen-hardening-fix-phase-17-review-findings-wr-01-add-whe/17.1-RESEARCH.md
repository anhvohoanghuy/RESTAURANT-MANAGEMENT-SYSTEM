**Researched:** 2026-07-09

## Summary

This is a hardening phase вАФ no new features. Every finding was verified against real code (Phase 15 REVIEW WR-01..05, Phase 17 REVIEW WR-01/02/03 + IN-01/02). The codebase is Spring Boot **4.0.6** (Jakarta, Spring 7, Jackson 3 `tools.jackson`), Spring for Apache Kafka **4.0.x**, Spring Data JPA on **MySQL** with `spring.jpa.hibernate.ddl-auto=update` (Hibernate auto-creates/updates tables вАФ **no Flyway/Liquibase**, so new `@Entity` classes materialize their tables automatically).

The seven mechanical fixes (whenComplete logging, fail-closed rank guard, existsByOrderId dedup, REQUIRES_NEW ledger reuse, rejection_reason TEXT + truncation, actorId audit, global serializer switch) are low-risk, localized diffs against named files. The one substantial item is **I-WR-02, the transactional outbox** for the three saga events. The lightest correct design for THIS codebase is: a single `outbox_events` JPA entity written in the SAME `@Transactional` method that mutates state (replacing the current `afterCommit` publish), plus a `@Scheduled` relay that claims pending rows with `SELECT ... FOR UPDATE SKIP LOCKED` (MySQL 8) and republishes them. No new dependencies вАФ plain `@Scheduled` + JPA + the existing Jackson-3 serde. Consumers are already idempotent (per-`eventId` ledgers), so at-least-once relay delivery is safe.

**Primary recommendation:** Implement one shared `outbox_events` table + `OutboxRelay` poller in the widest-reused location, store the event as pre-serialized JSON (byte-identical to the current wire format), and publish it verbatim via a `KafkaTemplate<String,String>` using `StringSerializer` вАФ because all consumers set `USE_TYPE_INFO_HEADERS=false` + a forced default type, the raw JSON round-trips identically. Do every other fix as a targeted diff to its named file. Keep all suites broker-free; run the full 156-test Maven suite after every change and especially after the global serializer switch.

### Recommended Project Structure (new files only; place per existing DDD layout)

### Pattern 1: REQUIRES_NEW ledger writer (I-WR-01 вАФ reuse the Phase 16 idiom verbatim)
**What:** Move the inline `saveAndFlush(ledger) + catch DataIntegrityViolationException` out of the saga `@Transactional` method into a separate `@Component` method annotated `@Transactional(propagation = REQUIRES_NEW)` that returns `boolean`.
**When to use:** `InventoryReservationService.onOrderCreated` (lines 80-92) and `OrderConfirmationService.onStockResult` (lines 53-62). Also candidate: `KitchenTicketCreationService` (44-53) and `KitchenStatusProjectionService` (65-74) share the identical anti-pattern.
**Example вАФ the pattern already exists, copy it:**
```java
// Source: src/main/java/.../inventory_context/application/InventoryLedgerWriter.java (Phase 16)
@Component
@RequiredArgsConstructor
public class InventoryLedgerWriter {
  private final InventoryProcessedEventRepository processedEventRepository;

  @Transactional(propagation = Propagation.REQUIRES_NEW)
  public boolean tryInsert(UUID eventId, String consumerName) {
    try {
      InventoryProcessedEventEntity ledger = new InventoryProcessedEventEntity();
      ledger.setEventId(eventId);
      ledger.setConsumerName(consumerName);
      ledger.setProcessedAt(Instant.now());
      processedEventRepository.saveAndFlush(ledger); // violation rolls back ONLY this inner tx
      return true;
    } catch (DataIntegrityViolationException duplicate) {
      return false; // caller's tx stays committable
    }
  }
}
// Caller becomes:  if (!ledgerWriter.tryInsert(eventId, CONSUMER_NAME)) return;
```
An equivalent `OrderLedgerWriter` (order_context) is the "extract an equivalent order-side writer" the CONTEXT mentions вАФ the Order context has no writer yet, only Inventory does.

**Critical rule:** the writer MUST be a distinct Spring bean (not a private method) or `REQUIRES_NEW` is bypassed by self-invocation (the existing Javadoc calls this out). The pre-check `existsByEventIdAndConsumerName` should remain as a fast path *before* calling the writer.

### Pattern 2: whenComplete send-failure logging (K-WR-01, I-WR-03)
**What:** `KafkaTemplate.send()` returns `CompletableFuture<SendResult<K,V>>` in Spring Kafka 3.x/4.x (migrated from `ListenableFuture` in 3.0). Attach `whenComplete` and log on `ex != null`.
**When to use:** all fire-and-forget adapters вАФ `KafkaKitchenSettleTriggerPublisher:20`, `KafkaKitchenTicketStatusChangedPublisher:22-23`, `KafkaInventoryStockResultPublisher:30`, `KafkaOrderEventPublisher:25,30`.
**Example:**
```java
// Add: private static final Logger log = LoggerFactory.getLogger(<Adapter>.class);
settleTriggerKafkaTemplate
    .send(settleTriggerTopic, event.orderId().toString(), event)
    .whenComplete((result, ex) -> {
      if (ex != null) {
        log.error("Settle trigger publish FAILED order={} line={} вАФ reservation may never settle",
            event.orderId(), event.orderLineId(), ex);
      }
    });
```
Note the scope split: for the **three outbox-owned events**, the outbox RELAY owns the `whenComplete` (it flips row status on success/failure). For any **remaining direct sends** вАФ the two Kitchen publishers (out of scope for the full outbox per CONTEXT) and any StockResult path not moved to the outbox вАФ apply the standalone `whenComplete` logging. Do not double-wrap: once an event is relayed via the outbox, its adapter no longer does the direct send.

### Pattern 3: `@Scheduled` outbox relay with SKIP LOCKED (I-WR-02)
**What:** poll loop that claims and publishes pending rows; row-level locking prevents double-publish across app instances.
**Example:**
```java
// Repository вАФ native query; MySQL 8 supports SKIP LOCKED
@Query(value = """
    SELECT * FROM outbox_events
    WHERE status = 'PENDING'
    ORDER BY created_at
    LIMIT :batch
    FOR UPDATE SKIP LOCKED
    """, nativeQuery = true)
List<OutboxEventEntity> claimPending(@Param("batch") int batch);

// Relay
@Component
@RequiredArgsConstructor
public class OutboxRelay {
  private final OutboxEventRepository repo;
  private final KafkaTemplate<String, String> outboxKafkaTemplate; // StringSerializer

  @Scheduled(fixedDelayString = "${outbox.relay.delay-ms:1000}")
  @Transactional
  public void poll() {
    for (OutboxEventEntity row : repo.claimPending(100)) {
      try {
        outboxKafkaTemplate.send(row.getTopic(), row.getMsgKey(), row.getPayload()).get(); // await inside tx
        row.setStatus("SENT"); row.setSentAt(Instant.now());
      } catch (Exception ex) {
        row.setAttempts(row.getAttempts() + 1);
        if (row.getAttempts() >= MAX_ATTEMPTS) row.setStatus("FAILED");
        // else leave PENDING for next poll (backoff via createdAt ordering)
      }
    }
  }
}
```
Requires `@EnableScheduling` on a config/app class (verify it isn't already enabled elsewhere вАФ grep before adding to avoid duplicate-annotation noise like the existing double `@EnableKafka`).

**Payload-as-string rationale:** store `objectMapper.writeValueAsString(event)` (or the byte[] from a `JacksonJsonSerializer`) at write time and publish it verbatim with a `StringSerializer`. Because every consumer config sets `USE_TYPE_INFO_HEADERS=false` and forces `VALUE_DEFAULT_TYPE`, the wire bytes are the JSON body only вАФ a re-serialized string is byte-identical to a direct `JacksonJsonSerializer` send. This avoids maintaining an event-typeвЖТtyped-template routing map in the relay. Serialize with the **Jackson-3 mapper** (Boot auto-configures a `tools.jackson` `JsonMapper`/`ObjectMapper` bean) so `Instant` serializes correctly вАФ do NOT hand-build a bare Jackson-2 mapper (the exact landmine I-WR-05 warns about).

### Anti-Patterns to Avoid
- **`getOrDefault(status, -1)` fail-open (K-WR-03):** `KitchenStatusProjectionService:95-96` lets `SUBMITTED`/`PENDING_CONFIRMATION` (rank -1) be overwritten by any fulfillment status, skipping `CONFIRMED`. Fix is exactly the review's snippet: `if (currentRank < 0 || targetRank <= currentRank) return;`.
- **Catching DIVE inside the business transaction (I-WR-01):** marks the whole tx rollback-only вЖТ `UnexpectedRollbackException` on commit. Always isolate via `REQUIRES_NEW`.
- **Self-invoking a `@Transactional(REQUIRES_NEW)` method:** proxy bypass вАФ must be a separate bean.
- **Fire-and-forget `send()`:** discards the `CompletableFuture`; failures invisible.
- **Deleting outbox rows in the relay:** prefer status flip (`SENT`) for auditability; a separate periodic purge of old `SENT` rows is optional and out of scope.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| At-least-once event delivery | Bespoke retry queue in memory | DB outbox row + `@Scheduled` relay | Memory queues die with the JVM вАФ the exact crash gap being fixed |
| Cross-instance poll de-dup | App-level distributed lock / leader election | `FOR UPDATE SKIP LOCKED` (MySQL 8) | Row locking is per-row concurrent and dependency-free |
| Ledger insert isolation | try/catch inside saga tx | `REQUIRES_NEW` writer bean (already exists) | Prevents rollback-only poisoning |
| JSON serialization of events | New hand-rolled `ObjectMapper` | Boot's Jackson-3 `JsonMapper` bean / existing `JacksonJsonSerializer` | Bare Jackson-2 mapper can't serialize `Instant` here |
| Send-result handling | Swallow the future | `whenComplete` | Standard `CompletableFuture` API |

**Key insight:** the transactional-outbox pattern is a solved problem; the value is applying it minimally (one table, one poller) rather than pulling in Modulith/Debezium/ShedLock. The repo already contains the two hardest sub-patterns (idempotent consumers + `REQUIRES_NEW` writer), so the outbox is mostly plumbing.

## Common Pitfalls

### Pitfall 1: `REQUIRES_NEW` bypassed by self-invocation
**What goes wrong:** extracting the ledger insert into a private method in the same class leaves it running in the caller's transaction вАФ the poisoning bug is not actually fixed.
**Why:** Spring's transaction proxy only intercepts calls through the bean reference.
**How to avoid:** separate `@Component` (as `InventoryLedgerWriter` already is); inject and call it.
**Warning signs:** a duplicate still throws `UnexpectedRollbackException` in tests.

### Pitfall 2: Outbox relay awaiting the send outside vs inside its transaction
**What goes wrong:** if the relay marks a row `SENT` before the broker actually acks, a crash loses the message (recreates the gap one layer up). If it never awaits, status can't reflect the result.
**Why:** `send()` is async.
**How to avoid:** block on the future (`.get()` with timeout) inside the relay tx before flipping status, OR use the `whenComplete` callback to update status in a follow-up tx. Simplest correct: await in-tx.
**Warning signs:** rows stuck `PENDING` forever, or `SENT` rows that never reached the topic.

### Pitfall 3: Global serializer switch is a no-op OR unexpectedly broad
**What goes wrong:** `application.properties:26` feeds only Boot's **auto-configured default** `KafkaTemplate`. All six existing publishers inject their own explicitly-defined typed templates, so **no current publisher reads this default** вАФ the switch is latent hygiene, not a behavior change for existing flows. The real risk is the opposite: if you *also* "align" Payment/Table by editing their own `PaymentKafkaProducerConfig`/`TableKafkaProducerConfig` (which still use Jackson-2 `JsonSerializer` and whose events DO carry `Instant`), you change live producers вАФ out of scope per CONTEXT (only line 26). Note those two are latent-broken against a real broker today but tests are broker-free so nothing catches it.
**How to avoid:** change ONLY line 26 to `...JacksonJsonSerializer`; leave Payment/Table configs alone; run the full suite to confirm green.
**Warning signs:** a Payment/Table test suddenly exercising real serialization failing вАФ would indicate scope creep.

### Pitfall 4: MySQL `SKIP LOCKED` version / dialect
**What goes wrong:** `FOR UPDATE SKIP LOCKED` requires MySQL **8.0+**; on 5.7 it is a syntax error.
**How to avoid:** confirm the deployed MySQL is 8.x (the JDBC URL and driver `com.mysql.cj` suggest 8, but the running server version is unverified). Fallback if 5.7: a claimed-flag update (`UPDATE ... SET status='CLAIMED', claimed_by=? WHERE status='PENDING' LIMIT n`) then select claimed rows.
**Warning signs:** relay throws SQL syntax error at first poll.

### Pitfall 5: `describe()` truncation must precede persistence, and account for the widened column
**What goes wrong:** even with `TEXT`, an unbounded concatenation is undesirable; and if TEXT isn't applied (older row width) a long string still overflows.
**How to avoid:** both вАФ set the column to TEXT (`@Column(columnDefinition = "TEXT")` or `@Lob`) AND cap `describe()` (e.g. limit listed shortfalls or `substring` with ellipsis to a safe length).

public class OutboxEventEntity {

  @Column(name = "aggregate_id",   nullable = false) private UUID   aggregateId;   // orderId

## Open Questions (RESOLVED during planning вАФ 2026-07-09)

**Resolution summary (see 17.1-05-PLAN.md):** (1) single shared `outbox_events` table in a shared/outbox infra package; (2) relay awaits sends in-tx via `future.get(timeout)`; (3) `@EnableScheduling` added once in a new `OutboxConfig` after grep-confirming it is not already present. All three recommended options were taken and are consistent across plans 05/06/07.

1. **One shared outbox table vs two per-context tables.** вАФ RESOLVED: single shared `outbox_events`.
   - What we know: the repo splits idempotency ledgers per context (`order_processed_events`, `inventory_processed_events`), suggesting per-context is house style; but a single generic `outbox_events` is less code and the events already cross package boundaries (Kitchen imports Inventory's `SettleTriggerEvent`).
   - What's unclear: how strictly context isolation is enforced for a pure-infrastructure table.
   - Recommendation: single shared `outbox_events` in a small shared/messaging infra package (least code, atomic-with-any-context write). If the planner wants strict isolation, mirror the ledger split into `order_outbox`/`inventory_outbox`. Either is correct; pick one and be consistent.

2. **Does the outbox relay await sends in-tx or via callback?**
   - Recommendation: await in-tx (`future.get(timeout)`) for simplest correctness; revisit only if throughput demands async.

3. **Is `@EnableScheduling` already present?**
   - Grep before adding to avoid a duplicate-annotation smell (the codebase already has double `@EnableKafka`, IN-03). Add it once on the app class or a single config.
