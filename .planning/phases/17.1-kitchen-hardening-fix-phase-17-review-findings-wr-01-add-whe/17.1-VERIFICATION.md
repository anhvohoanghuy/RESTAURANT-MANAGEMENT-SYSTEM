### CR-01 Assessment (explicit judgment requested)

**Question:** Is CR-01 (a) a genuine goal-blocking correctness gap for I-WR-01, or (b) an acceptable/pre-existing design tradeoff?

**Judgment: (a) — genuine goal-blocking correctness gap. Severity: BLOCKING.**

**Reasoning:**
- Verified directly in code (not just trusting REVIEW.md): all four REQUIRES_NEW-adopting consumers (`OrderConfirmationService.onStockResult`, `InventoryReservationService.onOrderCreated`, `KitchenStatusProjectionService.onTicketStatusChanged`, `KitchenTicketCreationService.onOrderConfirmed`) call `ledgerWriter.tryInsert(...)` — which does `processedEventRepository.saveAndFlush(ledger)` inside its own `REQUIRES_NEW` transaction, forcing an immediate commit — BEFORE the business mutation and `outboxWriter.save(...)` call in the outer `@Transactional` method.
- This means: ledger row commits first → business work + outbox write happen second → outer transaction commits third. Any failure between step 1's commit and step 3's commit (DB commit failure, `ObjectMapper.writeValueAsString` exception inside `OutboxWriter.save`, deadlock, connection reset, or the concrete unguarded NPE described in IN-01 for `KitchenTicketCreationService`) leaves the ledger row permanently committed while the business work is rolled back.
- On Kafka redelivery, the fast `existsByEventIdAndConsumerName` pre-check now returns `true`, so the consumer returns as a silent no-op. The `DefaultErrorHandler` retry/DLT safety net never engages because the retried delivery no longer throws — there is no automated recovery path.
- Concrete, named consequences per consumer (from REVIEW.md, confirmed plausible from the code read): an order can be stuck in `PENDING_CONFIRMATION` forever with no `OrderConfirmed`/`REJECTED` resolution; an inventory reservation verdict can be silently dropped, stalling the same saga; a kitchen ticket can simply never be created for a confirmed order; a fulfillment status snapshot can be silently dropped, leaving order status stale.
- The phase's own stated goal is "durable messaging" and "REQUIRES_NEW ledger idempotency" as robustness deliverables. As implemented, the REQUIRES_NEW pattern converts a self-healing failure mode (a concurrent-duplicate `DataIntegrityViolationException` that previously marked the outer transaction rollback-only — safely retried by Kafka redelivery) into a strictly worse, unrecoverable one (silent, permanent, operator-invisible event loss). This directly undermines the "durable messaging" intent that motivated I-WR-01 in the first place.
- The fact that `InventoryLedgerWriter` (Phase 16) already carried this same ordering defect before this phase does not make the defect "acceptable" in the context of this phase's own verification: 17.1 explicitly chose to *replicate* the identical flawed pattern onto three additional consumers as its I-WR-01 deliverable, tripling the blast radius of a defect that contradicts the phase's own stated durability objective, rather than fixing or flagging it.
- This is judged BLOCKING rather than minor because it defeats the core purpose of the requirement it claims to satisfy (idempotent, durable event processing) and the failure mode is silent — there is no log, metric, or alert that surfaces a stuck aggregate; an operator would only discover it via a customer complaint or manual DB inspection.

**This does not mean the phase's other deliverables are unsound** — the outbox mechanism itself (write-in-tx, relay-awaits-send-before-flip, crash-recovery re-publish) is correct and well-tested; the fail-closed rank guard is correct; the audit trail, dedup absorption, rejection_reason fix, and Jackson-3 switch are all correctly implemented and tested. The gap is narrowly scoped to the REQUIRES_NEW ledger-insert *ordering* relative to the business/outbox commit, common to all four ledger writers.

### Gaps Summary

The phase successfully delivers 6 of its 7 hardening work streams cleanly: whenComplete send-failure logging (K-WR-01/I-WR-03), advance audit trail (K-WR-02), same-order dedup absorption (K-IN-01/02), fail-closed status projection (K-WR-03), the transactional outbox mechanism itself (I-WR-02), rejection_reason overflow fix (I-WR-04), and the global Jackson-3 serializer switch (I-WR-05) — all verified against the actual codebase, all backed by green, broker-free unit/integration tests, and the full 206-test Maven suite passes with zero regressions.

The one blocking gap is the REQUIRES_NEW idempotency-ledger ordering defect (CR-01) affecting I-WR-01 across all four adopting consumers (Order confirmation, Order projection, Kitchen ticket creation, Inventory reservation). The literal must-have wording ("runs in its own REQUIRES_NEW transaction, never poisoning the outer transaction") is technically satisfied — the mechanism does isolate the concurrent-duplicate case as designed. However, the same mechanism creates a new, more severe correctness gap: a failure after the ledger row commits but before the outer business+outbox transaction commits results in silent, permanent, unrecoverable event loss with no operator-visible signal and no Kafka-redelivery recovery path. This directly undermines the phase's own stated "durable messaging" objective and is judged a genuine goal-blocking gap rather than an acceptable tradeoff, given the severity (permanently stuck orders/tickets/reservations) and silence (no error surfaces) of the failure mode, and given that this phase chose to replicate this exact pattern onto three new consumers as its I-WR-01 deliverable.

Recommended fix path (per REVIEW.md CR-01): move the idempotency-ledger insert into the same transaction as the business mutation + outbox write, inserting it last so a concurrent-duplicate unique-constraint violation rolls back the whole transaction (safe, self-healing via Kafka redelivery) rather than pre-committing a "processed" marker ahead of unguaranteed work. A companion fix (IN-01) should null-guard `KitchenTicketCreationService`'s iteration over `event.lines()`/`selectedToppings()`, since it is the most concrete way the CR-01 window is currently reachable in this codebase.
