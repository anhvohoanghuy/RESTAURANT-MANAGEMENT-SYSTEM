## Goal Achievement

### CR-01 / I-WR-01 Gap Closure Assessment

**Prior finding (BLOCKING):** All four idempotency-ledger-adopting consumers (`OrderConfirmationService.onStockResult`, `InventoryReservationService.onOrderCreated`, `KitchenStatusProjectionService.onTicketStatusChanged`, `KitchenTicketCreationService.onOrderConfirmed`) committed the `(eventId, consumerName)` ledger row in a separate `REQUIRES_NEW` transaction that ran and committed BEFORE the business mutation + outbox write in the outer transaction. A failure between the two commits left the ledger row permanently marking the event "processed" while the actual work was rolled back — silent, unrecoverable event loss with no Kafka-redelivery recovery path.

**Fix applied (commit ab84908):** The idempotency-ledger insert was moved out of the dedicated `REQUIRES_NEW *LedgerWriter` beans and inlined as the LAST statement of each consumer's own `@Transactional` method, executed after the business mutation and after `outboxWriter.save(...)`. Verified directly by reading all four files:

| Consumer | Ledger insert now | Ordering confirmed |
|----------|-------------------|---------------------|
| `OrderConfirmationService.onStockResult` (lines 53-93) | `recordProcessed(event.eventId())` called as step (4), after the CONFIRMED/REJECTED branch (which itself calls `outboxWriter.save(...)` on the CONFIRMED path) | LAST — same `@Transactional` method, no `REQUIRES_NEW` anywhere in the file |
| `KitchenStatusProjectionService.onTicketStatusChanged` (lines 63-117) | `recordProcessed(event.eventId())` called after `order.setStatus(target)`, as the final statement | LAST — same `@Transactional` method, no `REQUIRES_NEW` |
| `KitchenTicketCreationService.onOrderConfirmed` (lines 42-83) | `recordProcessed(event.eventId())` called after `kitchenTicketRepository.save(ticket)`, as step (3)/final statement | LAST — same `@Transactional` method, no `REQUIRES_NEW` |
| `InventoryReservationService.onOrderCreated` (lines 71-160) | `recordProcessed(eventId)` called after `outboxWriter.save(...)` (step 5), as step (6)/final statement | LAST — same `@Transactional` method, no `REQUIRES_NEW` |

Each `recordProcessed(...)` helper does a plain `processedEventRepository.save(ledger)` (no `saveAndFlush`, no separate transaction) — it now flushes and commits atomically with the outer transaction's commit. A concurrent-duplicate unique-constraint violation on that final insert now throws inside the outer `@Transactional` method, propagating and rolling back the WHOLE transaction (business mutation + outbox row + ledger row all undone together); Kafka redelivers, and the fast `existsByEventIdAndConsumerName` pre-check at the top of the method absorbs the replay as a cheap no-op. This is exactly the self-healing behavior CR-01 asked for.

**Artifacts deleted, confirmed by filesystem search:**

- `KitchenLedgerWriter.java` — no longer present anywhere in the repo (`find` returns nothing).
- `InventoryLedgerWriter.java` — still present, but its only remaining caller in `src/main` is `InventoryReservationSettlementService` (Phase 16, out of this phase's declared I-WR-01 scope — the original CR-01 gap and the four adopting-consumer artifact list in the prior VERIFICATION.md never named this service). `InventoryReservationService` (the phase-17.1 consumer) no longer references `InventoryLedgerWriter` at all — confirmed by grep, zero occurrences.
- No stray `@Transactional(propagation = REQUIRES_NEW)` annotations remain on any of the four fixed consumer methods — confirmed by grep for `propagation` across all four files: zero matches. The only remaining `REQUIRES_NEW` mentions in those files are prose comments describing the OLD (fixed) pattern, for future-reader context.

**IN-01 companion fix confirmed:** `KitchenTicketCreationService.onOrderConfirmed` now null-guards `event.lines()` (`if (event.lines() != null) { ... }`) and `toItem(...)` null-guards `line.selectedToppings()` (`if (line.selectedToppings() != null) { ... }`). A malformed `OrderConfirmedEvent` with null lines/toppings no longer NPEs after the ledger commit — it builds an empty-items ticket and completes normally. Confirmed by test `malformedEventWithNullLinesAndNullToppingsDoesNotThrowAndStillSavesTicket` (asserts `kitchenTicketRepository.save` called twice, no exception), green in the full suite.

**Test evidence of the behavioral change (not just mechanism):**
- `OrderConfirmationServiceTest.concurrentDuplicateLedgerInsertPropagatesToRollBackTheWholeTransaction` — stubs `processedEventRepository.save(any())` to throw `DataIntegrityViolationException`, asserts `service.onStockResult(...)` throws (propagates, does not swallow) — proves the whole transaction now rolls back on a duplicate instead of silently absorbing it in an isolated commit.
- `KitchenStatusProjectionServiceTest`, `KitchenTicketCreationServiceTest.concurrentDuplicateLedgerInsertPropagatesToRollBackTheWholeTransaction`, `InventoryReservationServiceTest.concurrentDuplicateLedgerInsertPropagatesToRollBackTheWholeTransaction` — same pattern, one per consumer, all green.

**Judgment: CR-01 / I-WR-01 gap is CLOSED.** The fix directly targets and eliminates the exact failure mode identified: ledger commit no longer precedes unguaranteed business work; a concurrent duplicate now safely rolls back the whole transaction instead of pre-committing a "processed" marker. The four consumers named in the original blocking gap are all fixed and each has a dedicated regression test proving the new rollback behavior. IN-01 (the concrete NPE trigger for the failure window) is also fixed with its own test. No override needed — this is a genuine code fix, not a scope deviation.

**Residual note (non-blocking, out of phase scope):** `InventoryReservationSettlementService` (Phase 16) still uses `InventoryLedgerWriter` with the original `REQUIRES_NEW`-commits-first ordering. This defect pre-dates Phase 17.1, was not part of the four consumers 17.1 added under I-WR-01, and was not named in the original CR-01 artifact list or the prior VERIFICATION.md's 21 truths. It is flagged here as an info-level anti-pattern for future-phase awareness, not as a gap against this phase's goal.
