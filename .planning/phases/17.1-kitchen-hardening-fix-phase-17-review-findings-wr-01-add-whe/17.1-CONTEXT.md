# Phase 17.1 — kitchen-hardening + inventory-messaging-hardening

**Type:** Inserted (URGENT) after Phase 17
**Depends on:** Phase 15, Phase 16, Phase 17
**Discussed:** 2026-07-09 (interactive, per-finding)
**Status:** Context gathered — ready to plan

## Goal

Close the outstanding robustness/quality debt logged during Phase 15 and Phase 17
code reviews across the **Kitchen** and **Inventory/Order** bounded contexts. All
findings were verified against real code before scoping. No new features — this is
hardening: durable messaging, audit trail, fail-closed guards, and idempotency
cleanup. Behaviour visible to API clients is unchanged except the new audit fields.

## Scope — decisions locked

### Kitchen context (from Phase 17 REVIEW.md)

- **K-WR-01 — durable settle-trigger send.** `KafkaKitchenSettleTriggerPublisher`
  and `KafkaKitchenTicketStatusChangedPublisher` currently call `send(...)` and
  ignore the returned `CompletableFuture`, so a failed send is silently lost and
  inventory never settles. **Fix:** attach `whenComplete()` that logs send failures
  (topic, key, cause). (This is the light mitigation; full outbox for kitchen is out
  of scope — the settle-trigger is best-effort re-drivable via re-advance.)
- **K-WR-02 — advance audit trail.** `KitchenTicketAdvanceService.advance(...)`
  accepts `actorId` but never uses it. **Fix:** persist `actorId` + advance
  timestamp on the item transition (on the ticket item, or an audit record) so there
  is a trail of who advanced each item.
- **K-WR-03 — fail-closed projection.** `KitchenStatusProjectionService` uses
  `FULFILLMENT_RANK.getOrDefault(status, -1)` (fail-open) at lines 95–96. **Fix:**
  on an unknown rank, log a warning and skip the transition (fail-closed) instead of
  allowing a jump that could skip CONFIRMED.
- **K-IN-01/02 — absorb same-order duplicate.** `KitchenTicketRepository.existsByOrderId`
  / `findByOrderId` are declared but never called; a same-order `OrderConfirmed`
  under a new eventId throws on the unique constraint and lands on the DLT. **Fix:**
  call `existsByOrderId` in `KitchenTicketCreationService` to absorb the duplicate
  gracefully (log + no-op) instead of routing to DLT.

### Inventory / Order context (from Phase 15 REVIEW.md — WR-01..05)

- **I-WR-01 — REQUIRES_NEW ledger idiom.** The original saga paths still use the
  inline `saveAndFlush` + `catch DataIntegrityViolationException` idempotency idiom
  (`InventoryReservationService.java:87-88`, `OrderConfirmationService.java:58-59`),
  which marks the surrounding transaction rollback-only on a duplicate. **Fix:**
  route the ledger write through the `REQUIRES_NEW` `InventoryLedgerWriter` pattern
  introduced in Phase 16 (extract an equivalent order-side writer if needed).
- **I-WR-02 — transactional outbox (major).** Saga events are published via
  `afterCommit`; a crash between DB commit and Kafka send permanently strands an
  order in `PENDING_CONFIRMATION`. **Fix (chosen — full outbox):** persist saga
  events (`OrderCreated`, `StockReservationResult`, `OrderConfirmed`) to an outbox
  table in the SAME transaction as the state change, plus a relay poller that
  publishes pending rows to Kafka and marks them sent. Replaces the `afterCommit`
  publish for these three event types. Idempotent relay (at-least-once → consumers
  already idempotent).
- **I-WR-03 — durable saga sends.** `KafkaInventoryStockResultPublisher.java:30`,
  `KafkaOrderEventPublisher.java:25,30` ignore the send future. **Fix:** if the
  outbox relay owns publishing, the relay handles failures/retries; any remaining
  direct sends get `whenComplete()` logging. Keep consistent with K-WR-01.
- **I-WR-04 — rejection_reason overflow.** `OrderEntity.rejection_reason`
  (`OrderEntity.java:39`) has no length → defaults to VARCHAR(255); `describe()`
  concatenates all shortfalls with no truncation → multi-ingredient rejection risks
  a `DataIntegrityViolationException` on MySQL. **Fix:** make the column `TEXT`
  (`columnDefinition`/`@Lob` or `length`) and/or truncate `describe()` output.
- **I-WR-05 — global Jackson-3 serde.** `application.properties:26` still sets the
  global producer `value-serializer` to Jackson-2 `JsonSerializer`, contradicting
  the phase's Jackson-3 decision (latent `Instant` serialization landmine).
  **Fix (chosen — global switch):** change the default value-serializer to the
  Jackson-3 serializer and re-run the FULL Maven suite to catch Payment/Table
  producer regressions.
