# Phase 17.1 Plan 01: whenComplete Kafka Send-Failure Logging Summary

**Attached whenComplete error-logging callbacks (topic + orderId key + cause) to all four fire-and-forget Kafka publishers so failed broker sends are logged instead of silently dropped, backed by a broker-free Mockito+Logback ListAppender test.**

## Performance

- **Duration:** 5 min
- **Started:** 2026-07-09T10:37:00+07:00
- **Completed:** 2026-07-09T10:41:32+07:00
- **Tasks:** 2
- **Files modified:** 5 (4 main + 1 new test)

## Accomplishments
- Every one of the four fire-and-forget saga/kitchen Kafka publishers (`KafkaKitchenSettleTriggerPublisher`, `KafkaKitchenTicketStatusChangedPublisher`, `KafkaOrderEventPublisher` publishOrderCreated + publishOrderConfirmed, `KafkaInventoryStockResultPublisher`) now attaches `whenComplete` and logs `ERROR` with topic, orderId key, and cause (plus orderLineId for settle-trigger and ticketId for ticket-status-changed) on a failed send — a failed send is now observable instead of invisible.
- None of the four adapters block (`.get()`) or rethrow; all `publish*` methods remain `void` and return immediately after attaching the callback.
- New broker-free `KafkaPublisherWhenCompleteTest` proves the failed-future path logs exactly one ERROR containing the orderId and does not throw, and the successful-future path logs nothing — for `KafkaKitchenSettleTriggerPublisher` and both `KafkaOrderEventPublisher` sends.
- Full Maven suite (195 tests) stays green after the change — no regressions.
