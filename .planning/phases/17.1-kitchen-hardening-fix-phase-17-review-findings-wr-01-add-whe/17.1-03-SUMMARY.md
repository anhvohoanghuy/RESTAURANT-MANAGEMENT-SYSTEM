# Phase 17.1 Plan 03: Fail-closed kitchen status projection Summary

**KitchenStatusProjectionService now fail-closes on an unknown current fulfillment rank (logs + skips) instead of fail-open getOrDefault(-1), and its idempotency ledger write moved to a new shared REQUIRES_NEW OrderLedgerWriter bean.**

## Accomplishments
- Created `OrderLedgerWriter` (`@Component`, `@Transactional(propagation = Propagation.REQUIRES_NEW)`), mirroring `InventoryLedgerWriter`, and wired `KitchenStatusProjectionService` to delegate its idempotency-ledger insert to it instead of an inline `saveAndFlush` + `catch(DataIntegrityViolationException)` block (I-WR-01, order side).
- Fixed the fail-open rank guard (K-WR-03): the guard now checks `currentRank < 0` first and fail-closes (logs a warning, skips the projection) before comparing `targetRank <= currentRank`. A pre-CONFIRMED order (`PENDING_CONFIRMATION`/`SUBMITTED`, rank absent from `FULFILLMENT_RANK`) can no longer be jumped straight to `PREPARING`/`READY`/etc. by an out-of-order kitchen snapshot.
- Forward transitions from a known rank (e.g. `CONFIRMED` → `PREPARING`) and non-advancing transitions (e.g. `READY` receiving a stale `PREPARING` snapshot) are unaffected — verified by new and pre-existing tests.

## Decisions Made
- Implemented the fail-closed guard as two sequential `if` statements rather than the single combined `if (currentRank < 0 || targetRank <= currentRank) { if (currentRank < 0) log.warn(...); return; }` snippet shown in RESEARCH.md, for readability; behavior and `grep -c 'currentRank < 0'` count (1) are identical to the plan's acceptance criteria.
- `OrderLedgerWriter` is deliberately a standalone `@Component` bean (not a private method) so Spring's transaction proxy applies `REQUIRES_NEW` correctly — required for `OrderConfirmationService` to reuse it in the Wave-2 follow-up plan (17.1-06).
