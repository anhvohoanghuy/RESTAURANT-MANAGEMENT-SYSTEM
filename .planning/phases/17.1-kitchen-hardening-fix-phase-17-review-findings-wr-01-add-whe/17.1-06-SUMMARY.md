# Phase 17.1 Plan 06: Order-side transactional outbox cutover Summary

Cut OrderSubmissionService and OrderConfirmationService over to the shared transactional outbox (I-WR-02, order side), adopted the REQUIRES_NEW OrderLedgerWriter for the confirmation-saga idempotency guard (I-WR-01), and widened + truncated the order rejection reason (I-WR-04).

## What Was Built

**Task 1 — OrderConfirmationService: REQUIRES_NEW ledger + OrderConfirmed to outbox + describe() truncation; OrderEntity TEXT column.**

`OrderConfirmationService.onStockResult` now:
- Keeps the fast `existsByEventIdAndConsumerName` pre-check, then delegates the authoritative idempotency insert to the injected `OrderLedgerWriter.tryInsert(eventId, CONSUMER_NAME)` (a `REQUIRES_NEW` Spring bean method from plan 03). A concurrent-duplicate unique-constraint violation is now absorbed inside that isolated transaction and returns `false` — the caller's transaction is never marked rollback-only.
- On the `CONFIRMED` branch, writes the `OrderConfirmedEvent` to the shared outbox in-transaction via the injected `OutboxWriter.save("ORDER", order.getId(), OrderConfirmedEvent.TYPE, orderConfirmedTopic, order.getId().toString(), toOrderConfirmedEvent(order))` instead of registering an `afterCommit` `TransactionSynchronization`. The `OrderEventPublisher` field, the `publishAfterCommit` helper, and the now-unused `TransactionSynchronization`/`TransactionSynchronizationManager` imports were removed.
- `describe(shortfalls)` now routes its result through a new `truncate(String)` helper that caps the string at `MAX_REASON_LEN = 60000` characters, appending a `"... (truncated)"` suffix when the cap is hit, so a rejection with a very large shortfall list can never overflow the persisted column.

`OrderEntity.rejectionReason` is widened from the implicit `VARCHAR(255)` to `@Column(name = "rejection_reason", length = 65535)`, which Hibernate maps to a `TEXT`-class column on MySQL (and a long column on H2 in tests) so `ddl-auto=update` applies the widening on next boot.

**Task 2 — OrderSubmissionService: OrderCreated to outbox in submit() tx.**

`OrderSubmissionService.submit(userId)` now writes the `OrderCreatedEvent` to the outbox via `outboxWriter.save("ORDER", saved.getId(), OrderCreatedEvent.TYPE, orderCreatedTopic, saved.getId().toString(), toEvent(response))` inside the existing `@Transactional` method, immediately before returning the response — instead of registering an `afterCommit` synchronization. The `OrderEventPublisher` field, `publishAfterCommit` helper, and the `TransactionSynchronization`/`TransactionSynchronizationManager` imports were removed.

Both services now depend on `OutboxWriter` (`com.example.feat1.DDD.shared.outbox.application`, `MANDATORY` propagation) built in plan 05, and `OrderConfirmationService` additionally depends on `OrderLedgerWriter` (`com.example.feat1.DDD.order_context.application`, `REQUIRES_NEW`) built in plan 03. `OutboxRelay` (plan 05) is the sole publisher of these rows to Kafka going forward, closing the crash-between-commit-and-send gap for both `OrderCreated` and `OrderConfirmed`.

## Test Changes

- `OrderConfirmationServiceTest`: reworked around mocked `OrderLedgerWriter`/`OutboxWriter` collaborators (field-injected `orderConfirmedTopic` set via `ReflectionTestUtils`). Covers: CONFIRMED → single `outboxWriter.save(...)` call with the correct aggregate type/id/eventType/topic/msgKey/payload (via `ArgumentCaptor`); REJECTED → no outbox write; the fast pre-check duplicate path (`existsByEventIdAndConsumerName == true`) never touches `ledgerWriter`/`orderRepository`/`outboxWriter`; a new test for the `tryInsert == false` concurrent-duplicate path asserting no order load and no outbox write; a new test with 2000 synthetic shortfalls asserting `rejectionReason.length() == MAX_REASON_LEN` (truncation actually caps the value); and the existing non-pending-order guard test. The obsolete `afterCommit`-synchronization test was removed since the write is now synchronous and in-transaction.
- `OrderSubmissionServiceTest`: constructor updated for the `OutboxWriter` dependency (`orderCreatedTopic` set via `ReflectionTestUtils`); the publish assertion now captures `outboxWriter.save(...)` args instead of `orderEventPublisher.publishOrderCreated(...)`, and the two negative-path tests assert `outboxWriter` was never called.
- `OrderSubmissionIntegrationTest`: dropped the `@MockitoBean OrderEventPublisher`; now autowires the real `OutboxEventRepository` + `ObjectMapper` and asserts a `PENDING` `OutboxEventEntity` row (`eventType == "OrderCreated"`, `msgKey == orderId`, `aggregateType == "ORDER"`) exists after a successful submit, deserializing the stored JSON payload back into an `OrderCreatedEvent` to assert on its contents. The empty-cart negative test asserts the pending-row count is unchanged instead of asserting a publisher was never called.
