# Phase 17.3 Research

## Research Complete

The issue is confirmed by local code inspection:

- `PaymentKafkaProducerConfig` imports `org.springframework.kafka.support.serializer.JsonSerializer` and sets `ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG` to `JsonSerializer.class`.
- `TableKafkaProducerConfig` imports the same legacy serializer and sets the same producer property.
- `application.properties` already sets the global producer value serializer to `org.springframework.kafka.support.serializer.JacksonJsonSerializer`, but these dedicated producer factories override that global setting with their own map.
- `PaymentEvent` and `TableOperationEvent` both contain `Instant occurredAt`, which matches the known classpath risk documented in existing `EventSerdeRoundTripTest`: Jackson-3 serde has native java.time support in this project, while the legacy path is the untested risk.

## Existing Patterns

- `OrderKafkaProducerConfig` explicitly configures `JacksonJsonSerializer.class`.
- Kitchen producer config tests instantiate producer config classes directly and inspect `DefaultKafkaProducerFactory#getConfigurationProperties()`.
- `EventSerdeRoundTripTest` uses broker-free `JacksonJsonSerializer` -> `JacksonJsonDeserializer` round trips with fixed `Instant` values and record equality assertions.

## Recommended Fix

Use one focused plan:

1. Replace legacy serializer imports/usages in Payment/Table producer configs with `JacksonJsonSerializer`.
2. Add broker-free producer config tests for both configs.
3. Add broker-free serde round-trip tests for `PaymentEvent` and `TableOperationEvent`.
4. Verify with focused tests, source scan, and full Maven suite.

## Scope Boundaries

- No dependency changes.
- No topic or event schema changes.
- No publisher behavior changes.
- No consumer/DLT/outbox changes.
